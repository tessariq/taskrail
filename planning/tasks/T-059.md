# T-059 Add read-only taskrail coverage command: coverage/orphan/drift analysis (--json)

## Description

Implement the deterministic subset of the v0.3.0 strategic sections тАФ Coverage
Checks, Orphan Task Checks, and Drift Detection тАФ as one strictly read-only
command, `taskrail coverage` (never writes STATE.md or task files). Reported
against the active spec: **coverage** (live active-spec headings with no linked
task, plus a coverage percentage), **orphans** (tasks whose `spec_ref` resolves
to an anchor outside the active spec тАФ the deterministic orphan rule; the looser
"otherwise detached from current intent" framing is intentionally *not*
implemented because it is not deterministic. A `spec_ref` with a non-existent
anchor stays a hard `validate` failure, not a soft signal), and
**drift** (the two-directional summary of uncovered spec areas and tasks
pointing away from the active spec). These are advisory signals: a coverage gap
must not make `validate` fail. Expose the coverage computation as a shared
internal capability that `stats` (T-057) reuses for its spec-coverage metric.
Spec-To-Task Decomposition and Gap Analysis stay out of scope (exploratory).
Spec: `specs/v0.3.0.md#deterministic-linkage-analysis`.

## Acceptance

- `taskrail coverage` reports coverage (with a percentage), orphan tasks, and
  drift against the active spec, and leaves the working tree clean (read-only).
- Coverage gaps and orphan/drift signals are advisory: `validate` still passes
  when they are present.
- `taskrail coverage --json` mirrors the report for agent consumption.
- The coverage percentage is exposed as a shared internal capability `stats`
  consumes (no duplicated computation).
- The coverable-area convention below is honored: roll-up, subsumption, and
  `> Deferred` exclusion all apply, and the dogfood spec `specs/v0.3.0.md`
  reports no false gap under it.
- With an active spec that has no coverable areas, coverage reports `N/A` (no
  percentage; JSON emits `null`) rather than 0% or 100%, and `validate` still
  passes.
- `README.md` and `CHANGELOG.md` are updated for the new command.
- `AGENTS.md` documents `coverage` as a read-only command (side-effect-free like
  `validate`), so agents know it needs no post-run `git status`/staging
  follow-up.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass.

## Coverable-heading convention

Mirror the spec contract (`specs/v0.3.0.md#deterministic-linkage-analysis`,
"Coverable-area convention"). Do not re-slug тАФ reuse `collectHeadingAnchors`;
coverage only narrows *which* anchors count as coverable areas.

- The coverable set is the feature areas under `## Potential Features`. Meta
  sections (Summary, Goals, Caution, Recommendation About LLM Support,
  Explicitly Excluded) are excluded so the percentage is not noise.
- **Roll-up.** A `###` area counts as covered when it *or any of its `####`
  sub-areas* has a linked task (for example *Skill Distribution Convergence* is
  covered through T-051/T-053/T-054/T-055, which link its `####` children).
- **Subsumption.** *Coverage Checks*, *Orphan Task Checks*, and *Drift
  Detection* are implemented by this command and covered by *Deterministic
  Linkage Analysis*; they are not separate uncovered areas.
- **Deferred.** A heading directly followed by a `> Deferred to <version>`
  marker (currently *Spec-To-Task Decomposition Support* and *Gap Analysis*) is
  excluded from the denominator.
- **No coverable areas.** When the active spec has no `## Potential Features`
  section, or that section resolves to zero coverable areas (as older specs like
  `v0.1.0`/`v0.2.0` may), the denominator is empty: report `N/A` with no
  percentage (JSON `null`, never `0`/`100`), so an unstructured spec is never
  scored as a false 0% or a hollow 100%. `--min` (T-066) treats `N/A` as passing
  тАФ there is nothing to gate on an unscoreable spec.
