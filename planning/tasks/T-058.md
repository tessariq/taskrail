# T-058 Distribute Windows build via WinGet using the GoReleaser winget publisher

## Description

Ship a Windows build and distribute it through WinGet, automated from the
existing tag-triggered GoReleaser pipeline. Use GoReleaser's built-in `winget`
publisher (open-source tier), not the standalone `winget-releaser` action. Add a
Windows build target (`windows/amd64` required, `windows/arm64` optional)
producing a `.zip`, and package as `InstallerType: zip` with
`NestedInstallerType: portable` â€” Taskrail is a single binary, so no MSI or
installer authoring is needed. Add a `winget` block:
`package_identifier: Tessariq.Taskrail`, publisher `Tessariq`, license, short
description, and `publisher_url`; point `repository` at a fork of
`microsoft/winget-pkgs` with `pull_request.enabled: true` and base
`microsoft/winget-pkgs`. Provide a classic PAT with `public_repo` scope as a
`WINGET_TOKEN` secret passed to the GoReleaser step (the default `GITHUB_TOKEN`
cannot open a cross-repository PR). Choose and document the first-submission
bootstrap (manual `wingetcreate` first vs GoReleaser opening the first PR).
Spec: `specs/v0.3.0.md#windows-distribution-via-winget`.

## Acceptance

- `.goreleaser.yaml` builds `windows/amd64` (arm64 optional) as a `.zip`;
  release assets include the Windows archive.
- A `winget` block produces a valid `Tessariq.Taskrail` manifest as
  zip+portable and is configured to open a PR against `microsoft/winget-pkgs`
  from the fork using `WINGET_TOKEN`. Task completion rests on the manifest and
  configuration verified by `goreleaser check`/snapshot â€” **not** on the live
  cross-repo PR (see completion boundary).
- `docs/workflow/releasing.md` documents the Windows assets, the `WINGET_TOKEN`
  secret, the fork prerequisite, the moderation-review latency, and the chosen
  bootstrap path.
- No MSI/NSIS and no self-hosted source; WinGet publication is coupled only to
  the existing tag-triggered release.
- `goreleaser check` passes; `gofmt -l .`, `go vet ./...`, `go test ./...`, and
  `taskrail validate` pass.

## Verification Notes

- `goreleaser release --snapshot --clean` (or `goreleaser check`) produces the
  Windows zip and a well-formed winget manifest; record the manifest and the
  dry-run PR body. The live winget-pkgs PR and moderation are validated at
  release time (needs `WINGET_TOKEN` and the fork).

## Completion boundary

- The live `microsoft/winget-pkgs` PR cannot be exercised in CI: it needs the
  fork, the `WINGET_TOKEN` secret, and Microsoft moderator review, none of which
  are self-controlled. This task is **done** when the Windows build, the winget
  block, the docs, and a green `goreleaser check`/snapshot manifest are in place.
- The actual cross-repo PR and its moderation are a release-time outcome tracked
  through `docs/workflow/releasing.md`; do not hard-gate this task's `verify` on
  the merged upstream PR.
