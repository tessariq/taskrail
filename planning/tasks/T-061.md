# T-061 v0.3.0 gap/drift — pre-release spec check

## Description

Pre-release gap/drift analysis for the **v0.3.0** release, mirroring T-049. Runs
after the version's implementation work lands and gates the release: v0.3.0 is
not tagged until this check passes. The spec is the source of truth
(`specs/v0.3.0.md`); do not duplicate spec content here.

## Release check

- Re-read the spec, focusing on **Goals** and **Potential Features**, plus the
  **Caution**, **Recommendation About LLM Support**, and **Explicitly Excluded**
  sections (drift guards).
- Walk the codebase against each intended feature/goal. A **gap** is a spec
  commitment not met; a **drift** is built behavior that diverges from the
  spec's intent (implemented differently, spec silently outgrown, or an
  Explicitly-Excluded item crept in).
- Once `taskrail coverage` (T-059) exists, run it as machine evidence for the
  coverage/orphan/drift half of the check. Read the **implementation-coverage**
  figure (T-080) as the "can we release?" signal: every coverable area should be
  implemented (all its non-cancelled tasks `completed`) before `--result pass`.
  Decomposition coverage is expected at 100% by this point; implementation
  coverage reaching 100% is the release-readiness evidence this gate records.
- Run the repo's stated validation as evidence: `gofmt -l .`, `go vet ./...`,
  `go test ./...`, `taskrail validate`; note results.

## Acceptance

- The spec is walked against the codebase; every gap and drift is recorded.
- Machine evidence (`taskrail coverage` where available, plus the validation
  suite) is captured.
- Each gap/drift that should enter the backlog becomes a follow-up task via
  `taskrail verify T-061 --create-followup ...` before the final transition.
- Closed with `taskrail verify T-061 --result pass|fail`; `--result pass` only
  when no unresolved gap or drift remains.

## Verification Notes

- `taskrail verify T-061` writes the artifact under `planning/artifacts/verify/`
  and updates `STATE.last_verification_result`; reference that evidence path.

## Implementation Notes

- This gate depends on every v0.3.0 task, including the stretch tasks
  (T-069, T-070, T-071, T-076, T-077), so it stays genuinely last: it cannot go
  eligible until all implementation — stretch included — is `completed`.
- Cutting a stretch task is therefore a **two-step** edit: `cancel` the task
  **and** remove its line from this task's `dependencies`. A `cancelled` dep does
  not satisfy `dependenciesResolved` (only `completed` does), so cancelling
  alone would leave a dep that never resolves and block this gate — and the
  release — permanently. Drop the dep line in the same change that cancels the
  task.
