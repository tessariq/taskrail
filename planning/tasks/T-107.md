# T-107 Make taskrail task rename atomic with rollback on partial write failure

## Description

`RenameTask` (added in T-096) performs its coupled writes sequentially — move the
file, rewrite the renamed task's frontmatter, rewrite each inbound dependent, then
re-project `STATE.md`. A failure after the first write (disk full, permission
revoked mid-run, process killed) leaves the tree partially renamed with no
compensating rollback, and the bare error does not steer the caller to
`validate`/`repair`. The feature is described as atomic ("all coupled edits as one
outcome"), so a partial write contradicts the contract. This mirrors the existing
non-transactional `saveAll` pattern (`Start`/`Unblock`/`Repair`), but rename is
riskier because it spans a filesystem rename plus several independent task files.

Follow-up derived from T-096's code review (go-reviewer MEDIUM finding).

## Acceptance

- A mid-run write failure in `RenameTask` either leaves the tree fully unchanged
  (rollback) or the emitted error explicitly directs the operator to
  `taskrail repair`/`validate` to reconcile.
- A test simulates a failure after the file move (e.g. an unwritable inbound task
  file) and asserts the resulting state is either the original or cleanly
  repairable — never a silently inconsistent `filename != id` tree.
- Add a real-`git init` integration test for `moveTaskFile`/`gitMove` (the T-096
  fixtures use a stub `.git` dir, so `git mv` always fails and only the plain-rename
  fallback is exercised). Cover both a `git mv` success (asserting the rename is
  tracked/staged, e.g. `git status --porcelain` shows `R`) and a `git mv` refusal
  (destination exists) — the latter is the exact path whose masking caused the
  T-096 CRITICAL finding. Guard with `exec.LookPath("git")`/skip when git is absent,
  mirroring `TestValidateCleanCheckoutOfCommittedRepo`.
