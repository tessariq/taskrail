# T-041 Make import --apply atomic via two-phase CreateTask validation

## Description

`taskrail import --apply` is not atomic. `ApplyImportDraft` writes the spec file,
then creates tasks one by one via `CreateTask`. `ValidateImportDraft` only checks
draft shape; `CreateTask` performs live-repo checks the validator does not
(`spec_ref` heading exists, external dependency exists, `spec_ref` non-empty). A
draft can therefore pass validation and still fail mid-apply, leaving a partial
write: the spec file plus any earlier tasks land on disk with no rollback. T-034
made that state visible (the error names what was written) but did not eliminate
it, and retrying is awkward because `writeImportedSpec` then refuses to overwrite
the orphan spec.

This task implements **option 1** from the T-034 review discussion: give
`CreateTask` a dry-run/validate-only mode and run all task validations up front —
accounting for headings the same apply is about to write — so nothing is written
unless the whole draft will succeed. A naive full pre-flight was rejected in T-034
because it breaks the legitimate `planning` case where a task's `spec_ref` points
at a heading in the spec that the same apply creates; the validate-only mode must
resolve those against the pending spec sections, not only the on-disk file.

Scheduled for the **end of v0.2.0**: this task depends on the other open v0.2.0
tasks (T-035, T-036, T-037, T-039, T-040) so `taskrail next` will not surface it
until they are done.

## Blocking Gate

- **Do not start implementation before re-interviewing the user.** The approach
  (two-phase validate vs. staging+atomic-rename vs. best-effort rollback), the
  desired retry/idempotency semantics, and whether atomicity is even wanted for a
  local single-user CLI are all open product decisions. Confirm scope and design
  with the user first, then proceed.

## Acceptance

- `CreateTask` (or an extracted validator) exposes a validate-only path that
  performs the live-repo checks without writing.
- `ApplyImportDraft` runs that validation for every task — resolving `spec_ref`
  headings against both existing spec files and the draft's own pending
  `spec_sections` — before any file is written; on any failure it writes nothing.
- The legitimate `planning` case (a task referencing a heading in the spec the
  same apply creates) still applies successfully.
- A draft whose task fails a live-repo check leaves the repository unchanged
  (no orphan spec, no partial tasks), demonstrated by a test.
- No behavior change to a fully valid apply.

## Verification Notes

- `go test ./internal/taskrail` and `go run ./cmd/taskrail validate`.
- Manual test the partial-failure and pending-spec-heading cases in a sandbox.
