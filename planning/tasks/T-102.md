# T-102 Document gap-analysis vs coverage boundary and mechanical-only limit

## Description

Gap analysis sits one word away from the shipped `coverage`/drift signals, and its
mechanical-only limit is easy to misread as "detect anything missing." Document the
boundary and the hard limit where operators meet them, per
`specs/v0.4.0.md#gap-analysis`, so nobody folds gap into coverage or expects semantic
inference from the binary.

## Acceptance

- Docs (README + relevant `docs/workflow/`) state the boundary plainly: `coverage`
  answers "is this spec area linked to any task?"; `coverage --gaps` answers "does a
  covered area lack a verification/companion task, have a graph anomaly, or look
  under-decomposed?".
- Docs state the hard mechanical limit: `--gaps` emits count/graph/state signals only,
  never semantic "this needs a test" inference (`specs/v0.4.0.md#explicitly-excluded`);
  false positives are expected and are candidates, not violations.
- Docs note `--gaps` is read-only and advisory by default, and that `--fail-on`
  (T-100) is the opt-in gate.
- Docs point operators at the `taskrail-gap` skill (T-101) for the semantic half.
