# T-020 Add guided retrofit detection for non-standard repositories

## Description

With the layout marker (T-018) and version-aware init (T-019) in place, extend
the retrofit flow so `init` can adopt a repository that does not match a known
Taskrail layout. It should analyze the repository, detect existing markers or
candidate directories (for example an existing `specs/` or notes/planning
folder), and propose a mapping for the adopter to confirm before writing
anything.

This is the analyze-and-adapt behavior the Guided Retrofit Bootstrap Flow in the
spec anticipates, made version-aware through the layout marker. Detection
proposes; it must never clobber existing files without explicit confirmation.

## Acceptance

- `init` (or a retrofit subcommand) detects an existing non-standard layout and
  proposes a mapping to the Taskrail layout instead of failing or silently
  overwriting.
- The proposed mapping is shown to the user and applied only on explicit
  confirmation; a dry run is the default.
- No existing file is overwritten or deleted without explicit confirmation.
- After applying, a `.taskrail/config.yml` marker records the resulting layout
  and version, and validation is re-run and reported.
- Unit tests cover detection of at least one non-standard layout and the
  no-clobber guarantee using temporary directories.

## Implementation Notes

- Build on the dry-run/diff/apply and marker primitives from T-018 and T-019;
  do not introduce a second, divergent confirmation mechanism.
- Keep detection conservative: prefer proposing a mapping the human confirms
  over inferring and applying automatically.
