# T-062 Add read-only spec list and spec show --anchors commands (--json)

## Description

Add the read-only half of the `spec` command family. `spec list` lists the
versioned specs under `specs/` and marks the active one. `spec show <version>`
prints a spec; with `--anchors` it lists the spec's live heading anchors exactly
as `spec_ref` validation computes them (reuse the same slug logic, not a
re-implementation), so `taskrail task new --spec-ref <path>#<anchor>` can be
authored against real anchors without reverse-engineering the slug rule. Both
are strictly read-only and support `--json`. In particular `spec show <version>
--anchors --json` emits a **stable, structured** anchor list (not scraped human
text) whose entries are exactly the anchors `spec_ref` validation accepts, so an
agent or the spec-command skill (T-064) can pick a real anchor from machine-clean
output; that stable shape is documented here so T-064 consumes it directly. Spec:
`specs/v0.3.0.md#spec-inspection-and-authoring-commands`.

## Acceptance

- `taskrail spec list` lists versioned specs and marks the active one;
  `--json` mirrors it.
- `taskrail spec show <version>` prints the spec; `taskrail spec show <version>
  --anchors` lists anchors identical to what `spec_ref` validation accepts.
- The anchor listing reuses the existing slug/anchor logic (a task authored from
  a listed anchor passes `validate`).
- `spec show <version> --anchors --json` emits a stable, structured anchor list
  identical to the anchors `spec_ref` validation accepts; a task authored from a
  JSON-listed anchor passes `validate`. The JSON shape is documented so T-064's
  skill consumes it directly (no scraping the human listing).
- Both commands are read-only (working tree stays clean).
- `README.md` is updated for the new commands; the `CHANGELOG.md` entry folds
  into the single `spec` commands entry shared across T-060/T-062/T-063 (one
  entry per user-facing feature, not one per command). Create that shared entry
  if it is absent and append to it if present, so these tasks landing in any
  order never produce a duplicate line.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass.

## Verification Notes

- In a sandbox, `spec show <version> --anchors`, author a task with one listed
  anchor, and confirm `validate` passes; assert `spec list` marks the active
  spec and the tree stays clean.
