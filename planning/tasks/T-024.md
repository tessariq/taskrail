# T-024 Stop validate from requiring gitignored artifact directories

## Description

`taskrail validate` requires `planning/artifacts` and `planning/artifacts/verify`
to exist as structural checks. In `internal/taskrail/service.go` the validate
required-directory list includes `ArtifactsDir` and `VerifyDir`.

Those directories are gitignored output, and `taskrail init` scaffolds them
empty. Git cannot track empty directories, so a normal commit/clone drops them,
and a freshly init-ed and committed repository fails `validate` on a clean
checkout (fresh clone or CI) for reasons unrelated to planning content. Because
v0.2.0 keeps artifacts deliberately gitignored, this is permanent: the verify
directory is empty on every clean checkout, so a "must exist" check can never be
satisfied by committed content.

A downstream project dogfooding v0.1.0 hit exactly this (CI red while local
pre-commit was green) and worked around it with committed `.gitkeep` files.

## Acceptance

- `validate` no longer requires `planning/artifacts` or `planning/artifacts/verify`
  to exist: remove `ArtifactsDir` and `VerifyDir` from the validate
  required-directory list in `internal/taskrail/service.go`.
- A missing artifacts tree is treated as "no artifacts yet" = valid; validate
  inspects artifact contents only when present.
- No placeholder files (`.gitkeep`) are introduced â€” the gitignored-artifacts
  decision stands.
- `taskrail verify` still creates `planning/artifacts/verify/<id>/<timestamp>/`
  on demand and writes `plan.md`, `report.json`, `report.md` (no change to
  evidence generation; keep a test asserting this).
- Invariant test: a freshly `taskrail init`-ed and committed repository passes
  `validate` on a clean checkout. Simulate with a temp dir: init, commit, then
  validate a worktree/clone containing only committed content; expect
  `{"valid": true, "violations": []}`.

## Verification Notes

- Run `go test ./internal/taskrail` and confirm the clean-checkout validate test
  and the verify-creates-dirs test pass.
- Manually reproduce in a temp sandbox: `init` in a fresh git repo, commit,
