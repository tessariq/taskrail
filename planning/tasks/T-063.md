# T-063 Add spec add <version>: scaffold spec file and update specs/README order

## Description

Add `spec add <version>`, the one writer in the `spec` family. It scaffolds
`specs/<version>.md` with the standard section skeleton (Summary, Goals,
Potential Features, Caution, Recommendation About LLM Support, Explicitly
Excluded) and updates the `specs/README.md` reading order. It creates a spec
file but never touches `STATE.md` or task state — activation stays a separate,
explicit step (`spec activate`, T-060). Reject a version that already exists or
does not follow the versioned-specs convention. Spec:
`specs/v0.3.0.md#spec-inspection-and-authoring-commands`.

## Acceptance

- `taskrail spec add <version>` creates `specs/<version>.md` with the standard
  skeleton and adds the version to the `specs/README.md` reading order.
- It never writes `STATE.md` or any task file; it does not activate the spec.
- An existing or non-conforming version is rejected with a clear error and no
  write.
- `README.md` and `CHANGELOG.md` are updated for the new command.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass
  (the scaffolded spec validates).

## Verification Notes

- In a sandbox, `spec add` a new version; confirm the scaffolded file, the
  updated README order, an unchanged `STATE.md`, and passing `validate`; attempt
  a duplicate version and confirm the error and no write.
