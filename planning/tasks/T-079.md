# T-079 Dogfood coverage --min in this repo's CI (gate on 100% self-coverage)

## Description

Wire this repository's own CI to `taskrail coverage --min` (T-066) so a future
spec area shipped without a linked task fails a push, not just the manual
pre-release gate (T-061). The v0.3.0 spec dogfoods to full self-coverage, so this
repo can gate on `coverage --min 100` against the active spec. The flag keeps
coverage advisory for `validate` (a gap never fails `validate`); the gate lives
only on the `coverage` command's exit code, invoked through the Taskfile target
convention T-074 formalizes and provisioned via mise. This is repo-internal
dogfooding, not a product invariant — adopters remain free to set their own
threshold or omit the gate. Spec:
`specs/v0.3.0.md#deterministic-linkage-analysis`.

## Acceptance

- A Taskfile target runs `taskrail coverage --min <pct>` against the active spec,
  and `.github/workflows/ci.yml` invokes it on each push and pull request
  (through the Taskfile target, per T-074).
- The threshold is chosen against the current self-coverage (100% while the spec
  dogfoods clean); the gate fails CI when coverage drops below it.
- `validate` stays unaffected: a coverage gap still never makes `validate` fail;
  only the new `coverage --min` step gates.
- The `N/A` case (a spec with no coverable areas, per T-059) is treated as
  passing, so activating a freshly-scaffolded spec does not red the gate.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass.

## Verification Notes

- Run the Taskfile target locally under mise; confirm it exits zero at current
  coverage and non-zero when the threshold is set above the actual percentage.
