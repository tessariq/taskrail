# T-052 Audit taskrail repair for wider deterministic cases; specify the safe set

## Description

Audit tracked-work inconsistencies and specify — without changing repair code —
the set of additional mechanical cases beyond the v0.2.0 set (a `current_task`
pointer disagreeing with the in_progress task, a stale `current_task_title`,
stale rendered counts) that `taskrail repair` could safely fix. A case qualifies
only when it is genuinely deterministic: exactly one safe correction exists, and
applying it neither advances a status nor fabricates work. Evaluate the spec's
candidates — stale focus/summary fields, other rendered STATE.md body
projections, and precisely reportable dangling references (report, never invent
the target) — and any others the audit surfaces. The deliverable is a concrete,
reviewable enumeration each entry naming the STATE.md field, the drift, and the
single safe correction, plus the boundary cases that stay human-resolved. The
implementation of the qualifying cases is T-072. Spec:
`specs/v0.3.0.md#extended-state-repair`.

## Acceptance

- A written, reviewable enumeration of the additional mechanical repair cases is
  produced: each qualifying case names the STATE.md field, the drift condition,
  and the single provably-safe correction (no status advance, no fabrication).
- The hard-boundary cases — a missing `spec_ref`, a dependency cycle, and more
  than one in_progress task — are listed explicitly as staying human-resolved,
  with the reason each admits more than one safe resolution.
- The enumeration is concrete enough that T-072 can implement it without
  re-deciding scope; it is captured in a form T-072 consumes (the task hand-off).
- This task changes no repair behavior; existing checks stay green
  (`gofmt -l .`, `go vet ./...`, `go test ./...`, `taskrail validate`).

## Verification Notes

- Capture the case enumeration as the task's hand-off to T-072 and reference it
  from the verification record; confirm no repair code changed under this task.
