# T-052 Extend taskrail repair to wider mechanical cases (keep no-fabricate boundary)

## Description

Widen `taskrail repair` beyond the v0.2.0 set (a `current_task` pointer
disagreeing with the in_progress task, a stale `current_task_title`, stale
rendered counts) to every additional case that is genuinely deterministic:
exactly one safe correction exists, and applying it neither advances a status
nor fabricates work. Audit tracked-work inconsistencies and evaluate candidates
such as stale focus/summary fields, other rendered STATE.md body projections,
and precisely reportable dangling references (report, never invent the target).
Keep the hard boundary: a missing `spec_ref`, a dependency cycle, and more than
one in_progress task stay human-resolved. Repair still only ever rewrites
STATE.md, defaults to a dry run showing the diff, and re-validates after
`--apply`. Spec: `specs/v0.3.0.md#extended-state-repair`.

## Acceptance

- Repair covers the additional mechanical cases surfaced by the audit, each with
  a single provably-safe correction; dry run shows the diff and `--apply`
  re-runs validation.
- Repair never rewrites a task file, never advances a status, never fabricates
  work.
- A missing `spec_ref`, a dependency cycle, and more than one in_progress task
  remain unrepaired and are reported through validation.
- Tests cover each new mechanical case and each untouched boundary case.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass.

## Verification Notes

- Seed sandboxes with each new mechanical drift and each boundary case; run
  `repair` (dry run then `--apply`) and record diffs plus re-validation output.
