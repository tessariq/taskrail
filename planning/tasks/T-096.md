# T-096 Add taskrail task rename: atomic re-slug of id, filename, and inbound dependency refs

## Description

There is no first-class way to re-slug an existing task. Doing it by hand couples
three edits — frontmatter id, filename, and every inbound `dependencies:` reference —
and missing the last silently breaks dependency resolution. Add `taskrail task rename
<id>` to do all of it atomically, per `specs/v0.4.0.md#task-rename-and-re-slug`.

## Acceptance

- `task rename <id> --slug Y` (or `--title "…"`, deriving the slug via the same
  `slugify` as T-095) rewrites the `id:` frontmatter, renames the file to
  `<new-id>.md` (`git mv` under version control, plain rename otherwise), rewrites
  every inbound `dependencies:` reference from the old id to the new id, re-projects
  `planning/STATE.md`, and re-runs `validate` — all as one outcome.
- After rename, `validate` passes and no dependency edge is left dangling.
- Exactly one of `--slug` / `--title` applies; the numeric `T-<n>` is preserved and
  only the slug segment changes.
- A target id colliding with an existing task makes no partial change and fails
  validation cleanly.
- `--dry-run` reports the frontmatter id change, file rename, and each inbound
  dependency edit, and writes nothing (mirrors `repair`).
- `--json` emits the planned or applied change set.
