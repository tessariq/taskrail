# T-089 Guidance: handling a conflicting STATE.md across parallel PRs

## Description

When several PRs are in flight, each rewrites the generated `planning/STATE.md`
(a projection of the task files), so branches collide on it and every rebase
retriggers CI. Today the contract tells agents to commit the CLI-regenerated
`STATE.md`, but says nothing about what to do when a merge/rebase *conflicts* on
it — so agents hand-resolve the derived file, which is both error-prone and
exactly what the "never hand-edit `STATE.md`" rule forbids.

Provide explicit, agent-facing guidance for resolving a conflicting `STATE.md`.
The through-line: `STATE.md` is a projection of the task files (the source of
truth), so a conflict on it is never hand-merged — the conflict is *sidestepped*
and the file is *regenerated*. `taskrail repair --apply` re-projects a correct
`STATE.md` from the merged task files, which merge cleanly because each PR edits
its own task file. This is the same mechanical-reconciliation contract from
`specs/v0.3.0.md#extended-state-repair`, extended to the git-merge scenario.

Guidance should cover the merge-conflict resolution flow (take either side, then
`repair --apply` to rebuild from task files; never hand-edit conflict markers in
`STATE.md`), the boundary repair will not cross (a genuine conflict on the *same
task file* is a real content conflict for a human — repair only re-projects the
aggregate, and still refuses ambiguous cases like more than one in_progress
task), and the optional repo-level hardening that removes the conflict at the
source (a `merge=ours` merge driver on `planning/STATE.md`, or treating it as
derived and regenerating it). It must not change any CLI behaviour or the spec
contract — this is documentation plus skill delivery only.

Deliver it as part of the packaged skills so coding agents pick it up
automatically: extend `internal/taskrail/skills/taskrail-repair/` (the single
source of truth) with a conflict-resolution section, then regenerate the
committed `.agents/`/`.claude/` copies with `task skills:regen` so the
package-parity check stays green. Spec:
`specs/v0.3.0.md#extended-state-repair`.

## Acceptance

- The `taskrail-repair` skill under `internal/taskrail/skills/` gains explicit
  guidance for a conflicting `STATE.md`: resolve by taking either side and
  running `${TASKRAIL:-taskrail} repair --apply` to re-project from the task
  files — never hand-edit conflict markers in `STATE.md`.
- The guidance states the boundary: repair only re-projects the derived
  aggregate; a real conflict on the *same task file*, or an ambiguous state repair
  refuses (e.g. more than one in_progress task), stays human-resolved.
- The guidance mentions the optional at-the-source hardening (a `merge=ours`
  driver on `planning/STATE.md`, or treating `STATE.md` as derived/regenerated)
  as a repo-level choice, without mandating a change to Taskrail CLI behaviour.
- No CLI behaviour or spec contract changes; this is documentation and skill
  content only.
- The committed `.agents/`/`.claude/` skill copies are regenerated from the
  package (`task skills:regen`) and byte-identical to it (`task check:skills`
  passes).
- `AGENTS.md`'s tracked-work/commit guidance references the conflict-resolution
  flow (or points at the skill) so the repo's own contract is consistent with
  the packaged advice.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass.
