# T-087 Reconcile block status_summary when another task is in_progress

## Description

Discovered while implementing T-086 (`taskrail unblock`). `block` routes through
`finishTask` (`internal/taskrail/transitions.go:362`), whose `blocked` branch
sets `state.status_summary = "blocked"` unconditionally. When a *different* task
is already `in_progress` (so `current_task` is set), blocking a todo task leaves
`status_summary` reading `blocked` even though the active task is still in
progress — a snapshot that contradicts `current_task`. `repair` does not
reconcile this direction (`internal/taskrail/repair.go:108`), so the drift is not
self-healing.

`unblock` (T-086) already guards this case: it only rewrites
`status_summary`/`next_action` when `current_task == ""`
(`internal/taskrail/transitions.go`), leaving an active task's pointers intact.
`finishTask`'s `blocked` branch should adopt the same guard so `block` never
clobbers an active task's summary.

Follow-up derived from T-086's verification or discovery.

## Acceptance

- Blocking a task while a different task is `in_progress` leaves
  `status_summary` = `in_progress` and preserves the active task's `next_action`;
  the newly blocked task's reason is still recorded in the `blockers` list.
- Blocking a task with no active task keeps today's behavior
  (`status_summary` = `blocked`, `next_action` = `Resolve blocker on <id>`).
- Mirror the `current_task == ""` guard `unblock` uses; do not special-case
  `complete` (completing the active task clears `current_task` first, so its
  idle reset stays correct).
- Table-driven unit test covers block-while-another-in-progress (summary
  preserved) and block-while-idle (unchanged). TDD: start with a failing test.
