# T-072 Implement widened repair cases from the T-052 audit

## Description

Implement the additional mechanical repair cases enumerated by the T-052 audit,
widening `taskrail repair` beyond the v0.2.0 set. The qualified-case enumeration
is delivered by T-052 into this task's `## Implementation Notes`; implement
exactly that set and no more. Each implemented case must be one the audit proved
deterministic: a single safe correction that neither advances a status nor
fabricates work. Repair still only ever rewrites STATE.md (never a task file),
defaults to a dry run showing the diff, and re-runs validation after `--apply`.
The hard boundary from the audit stays intact — a missing `spec_ref`, a
dependency cycle, and more than one in_progress task remain human-resolved and
are reported through validation, never auto-fixed. Spec:
`specs/v0.3.0.md#extended-state-repair`.

### T-052 Audit Hand-off: Qualified Widened Repair Cases

Audit of `internal/taskrail/repair.go`, `validation.go`, `transitions.go`, and
`templates.go` (`renderStateBody`) against the v0.3.0 spec's Extended State
Repair candidates. **Qualification test** (all three must hold): the drift is
detectable from committed files; exactly one correction exists; applying it
touches only `STATE.md` frontmatter and neither advances a task status nor
fabricates work.

**Repair's existing shape (what T-072 extends).** `Repair` (`repair.go:42`)
already: (1) reconciles the `current_task`/`current_task_title` pointer to the
single in_progress task via `repairCurrentTask`, and (2) recomputes the **entire
STATE.md body** with `renderStateBody(corrected, tasks)` and diffs it
(`repair.go:50-51`). It writes frontmatter + body, never a task file.

#### A. Already covered by the v0.2.0 set — do NOT re-implement

- `current_task` pointer disagreeing with the in_progress task
  (`repairCurrentTask`, `repair.go:79`). Includes the no-in_progress case
  (clears the pointer) and the >1 in_progress guard (refuses to act).
- `current_task_title` stale — reconciled independently of the pointer in the
  same function (`repair.go:99-102`), so a correct pointer with a renamed title
  is already fixed.
- **Body-vs-source consistency for the whole body**, because it is re-rendered
  wholesale from `(corrected frontmatter, tasks)` (`templates.go:138-192`). Note
  the precise guarantee: the re-render makes each body line match its *source*, it
  does not correct a source that is itself wrong. Only two body sections are
  sourced from **task state** rather than a frontmatter echo, so only these are
  reconciled against task-file truth with no new code:
  - **Current Focus** — sourced from `CurrentTask`/`CurrentTaskTitle`, which
    `repairCurrentTask` has already corrected before the render (`repair.go:49`).
  - **Task Counts** — computed directly from `tasks` by `taskCounts`
    (`templates.go:194`), no frontmatter intermediary.
  The remaining body sections (Active Spec, Status line, Blockers, Last
  Verification, Next Action, Relevant Artifacts, Notes) are verbatim echoes of the
  like-named frontmatter fields; the re-render keeps the body equal to those
  fields but leaves a wrong field wrong — those are addressed by QC-1 (B) or
  excluded (D), not here. The spec candidate "other rendered `STATE.md` body
  fields that are pure projections of task state" thus needs **no new body-level
  case**: the genuinely task-derived ones (Focus, Counts) are already reconciled.

#### B. Qualifying NEW case to implement

- **QC-1 — `status_summary` frontmatter, in_progress direction.**
  - Field: `status_summary` (frontmatter only; the body Status line is a verbatim
    echo of it, so — like every frontmatter echo — a wrong value survives the body
    re-render. `status_summary` is the one such echoed field that has a
    task-derivable correct value, which is why it qualifies where the others in
    (D) do not).
  - Drift: exactly one task file has `status: in_progress` but frontmatter
    `status_summary != "in_progress"`.
  - Single safe correction: set `status_summary = "in_progress"`.
  - Proof of safety: `status_summary` is a closed vocabulary
    `{"idle","in_progress","blocked"}` (seed `templates.go:17`; writers
    `transitions.go:96,373,376`). `Start` is the *only* writer that sets it while
    a task is in_progress (`transitions.go:96`); `verify` leaves it untouched. So
    "one in_progress task ⟹ `status_summary == "in_progress"`" holds by
    construction — the corrected value is unique. The fix rewrites only
    `STATE.md` frontmatter; task files (the status source of truth) are not
    touched, so no status advances and nothing is fabricated. Scope it under the
    same `len(inProgress) > 1 → refuse` guard `repairCurrentTask` already honors.

#### C. Report-only — surface through validation, never auto-fix (already wired)

The spec's third candidate — "dangling references that can be *reported*
precisely (repair surfaces them; it does not invent the missing target)" — is
**already satisfied without new code**: `Repair` returns `Validation`
(`repair.go:67-71`), and validation already flags committed references to
gitignored artifact paths in state frontmatter (`stateArtifactRefs`,
`validation.go:140`) and in task files (`taskArtifactRefs`, `validation.go:164`).
Per the spec these are *reported, not repaired* — the non-artifact text cannot be
invented — so T-072 must not add an auto-clear for them.

#### D. Considered and excluded (not deterministic — keep out of scope)

- `status_summary`, the **idle/blocked direction**: when no task is in_progress,
  the value ("idle" vs "blocked") reflects the *last transition*
  (`transitions.go:373,376`), not current task-file state — a blocked task can
  persist while later work returns the summary to "idle". Two safe values exist
  ⇒ ambiguous ⇒ excluded. (A stale `"in_progress"` with zero in_progress tasks is
  wrong but its replacement is likewise idle-vs-blocked ambiguous, so it also
  stays out.)
- `blockers` list: set from the authored block reason (`transitions.go:375`) and
  not reconstructable from task files; cleared by later transitions independent
  of blocked tasks remaining ⇒ no deterministic projection.
- `next_action`: advisory hint; recomputing it *is* the job of `taskrail next`,
  and validation asserts no invariant on it ⇒ not repair's concern.
- `last_verification_result` / `continuation_notes`: history/authored prose, not
  projections ⇒ report-only (dangling-artifact scan in C), never regenerated.
- `active_spec_version` vs `active_spec_path`: reconciling them relies on the
  filename convention, not task state, and `taskrail spec activate` (T-060) is
  the sanctioned writer ⇒ excluded.
- `schema_version`: a mismatch signals a schema *migration*, not mechanical
  drift; forcing it to the current constant would mask a genuine version
  incompatibility ⇒ excluded.

#### E. Hard boundary — stays human-resolved (report through validation, never fix)

- **Missing `spec_ref`** (`validation.go:288`): repair cannot invent a spec
  linkage; more than one spec/anchor could be correct.
- **Dependency cycle** (`detectDependencyCycles`, `validation.go:182`): repair
  cannot choose which edge to cut; every cut is one of several valid resolutions.
- **More than one in_progress task** (`validation.go:307`): repair cannot know
  which is authoritative; picking one would regress another's status (and would
  itself require editing a task file, which repair never does).

**Net for T-072:** implement exactly QC-1 (one new frontmatter case). Everything
else is already covered (A), already surfaced (C), or provably out of scope
(D, E). This changes no task-file behavior and keeps the "never advance status or
fabricate work" invariant absolute.
