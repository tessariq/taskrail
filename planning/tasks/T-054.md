# T-054 Ship autonomous-recovery and autonomous-manual-test in the packaged skill set

## Description

Add `autonomous-recovery` and `autonomous-manual-test` to the packaged
(embedded) skill set so the shipped set becomes a superset this repository can
adopt without losing capability. `autonomous-recovery` already routes every
correction through `taskrail repair` and forbids hand-editing authoritative
state; shipping it needs the entry-point indirection (T-051) and the implemented
extended repair (T-072) shrinking its human-resolved residue. Ship
`autonomous-manual-test` as a repo-agnostic skill, but keep its artifacts
ephemeral and gitignored: manual-test is NOT promoted to a product invariant —
`init` must not provision `planning/artifacts/manual-test/` and `validate` must
stay unaware of it. Update the productization doc disposition (recovery:
dogfooding-only -> shippable). Spec:
`specs/v0.3.0.md#ship-autonomous-recovery-and-autonomous-manual-test`.

## Acceptance

- `internal/taskrail/skills/` contains repo-agnostic `autonomous-recovery` and
  `autonomous-manual-test` that invoke `${TASKRAIL:-taskrail}`; both install via
  `init --with-skills`.
- manual-test artifacts stay gitignored; `init` does not create the manual-test
  directory and `validate` does not require it.
- `docs/workflow/skills-productization.md` marks recovery shippable and records
  the manual-test ship-skill / ephemeral-artifacts decision.
- Shippable-skills tests cover the two added skills.
- `CHANGELOG.md` gains a single Unreleased skills entry — shared with T-064 —
  noting the expanded packaged skill set (`autonomous-recovery`,
  `autonomous-manual-test`, and the spec-command skill), one entry per
  user-facing feature rather than one per skill.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass.

## Verification Notes

- `init --with-skills` in a sandbox lists both skills; confirm no manual-test
  directory is provisioned and `validate` passes without it.
