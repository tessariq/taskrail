# T-094 Surface empty-slug and duplicate-slug ### area headings as a validate/coverage diagnostic

## Description

Follow-up from T-093. T-093 fixed the `coverage --area` request boundary to
reject empty-slug and duplicate-slug `###` anchors, but deliberately left the
full-report denominator unchanged. Two degenerate `###` headings under
`## Potential Features` therefore still distort the aggregate figures reported by
`coverage`, `status`, and `stats`:

- A punctuation-only `###` title slugs to `""` (via `slugHeading`) yet still
  parses to a `parsedArea{anchor: ""}` and counts toward `CoverableAreas`.
- Two `###` headings that slug to the same anchor both count, inflating the
  denominator by counting one logical area twice.

Concretely, a spec with `### !!!`, `### Dup Area`, `### Dup Area`, `### Alpha`
(one covered) reports `1/4 = 25%` where a human reads two real areas.

This task chooses **Option B** from the T-093 review: rather than silently
dropping these headings from the denominator (which could hide a typo'd real
feature area), surface them as a **diagnostic** so the spec author is told to fix
the heading. Decide precisely where the diagnostic belongs — a `validate`
violation vs. an advisory `coverage`/`stats` signal — keeping in mind the
existing contract that coverage signals never make `validate` fail
(`specs/v0.3.0.md#deterministic-linkage-analysis`). `validate` already dedupes by
slug and skips empty slugs when accepting `spec_ref`s (`collectHeadingAnchorList`
in `internal/taskrail/validation.go`); align with, don't contradict, that
leniency.

## Acceptance

- An empty-slug `###` area heading (punctuation-only title) and a pair of `###`
  headings that slug to the same anchor are surfaced by a clear diagnostic that
  names the offending heading(s), rather than silently inflating the coverage
  denominator.
- The chosen surface is documented and internally consistent: if it is a
  `validate` violation, it does not break the "coverage signals never fail
  validate" contract; if it is an advisory `coverage`/`stats` signal, it does not
  gate. State which and why.
- `coverage`, `status`, and `stats` remain read-only; no state or task file is
  written by the new diagnostic.
- Existing non-degenerate specs are unaffected: a spec with unique, non-empty
  `###` area slugs reports exactly the same figures as before.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass.

## Verification Notes

- In a sandbox, author a spec with a punctuation-only `###` title and a same-slug
  `###` pair; assert the diagnostic fires and names the headings, that the
  aggregate figures are no longer silently distorted per the chosen design, and
  that the working tree stays clean (read-only). Store evidence under
  `planning/artifacts/manual-test/T-094/<timestamp>/`.
