# T-051 Configurable skill entry point (${TASKRAIL:-taskrail}); repo resolves the fallback via a mise-built binary

## Description

Shipped skill text must call the Taskrail binary through an overridable
indirection, `${TASKRAIL:-taskrail}`, instead of a hardcoded `taskrail` or
`go run ./cmd/taskrail`. Adopting repositories need nothing: the default
resolves to the installed `taskrail` binary.

This repository dogfoods the working tree **without setting a `TASKRAIL` env
override**. An env var only resolves under some activation mechanism
(mise-activate, direnv, a shell rc) that agent shells routinely skip ŌĆö and
`.claude/settings.json` env would be Claude-Code-specific ŌĆö so a bare
`${TASKRAIL:-taskrail}` in an unactivated agent shell would silently fall back to
the **installed v0.2.0 binary** and fail on the new v0.3.0 subcommands. Instead,
make the *fallback itself* correct: this repo builds the working-tree binary and
exposes it as `taskrail` on the **mise-provided PATH** (the wiring lands in
T-074's setup/Taskfile). Then plain `taskrail` resolves to the working tree in
any shell and any runner (Claude, `.agents/`, humans, CI) with no env var and no
Claude-specific config. This reuses the single PATH contract the repo already
mandates for `go`/`task`/`lefthook`, so it adds no new precondition, and it turns
silent staleness into a **total, loud** failure: if the mise PATH is absent, `go`
and `task` are gone too, so nothing runs at all rather than `taskrail` silently
resolving stale. The indirection stays overridable, so an adopter who *wants*
`TASKRAIL=/path/to/taskrail` (or `go run`) can still set it. This removes the
reason the dogfooding and shipped skills diverged and unblocks the repo adopting
the packaged set (T-055). Spec: `specs/v0.3.0.md#configurable-skill-entry-point`.

## Acceptance

- Every shipped skill under `internal/taskrail/skills/` invokes the binary as
  `${TASKRAIL:-taskrail} <args>`, with no hardcoded `taskrail` or
  `go run ./cmd/taskrail`.
- `mise.toml` sets **no** `TASKRAIL` env override; the fallback is correct by
  construction (the mise-built binary from T-074), not by an env var.
- In a plain, non-activated shell with the mise PATH present (as T-074 provides),
  bare `taskrail` resolves to the working-tree build: running a shipped skill's
  command exercises v0.3.0 behavior, not the stale installed binary ŌĆö with no env
  var and no `.claude`-specific config.
- With no mise PATH and nothing set, `${TASKRAIL:-taskrail}` resolves to the
  installed `taskrail` (the adopter default), and an explicit `TASKRAIL=...`
  override is still honored (the indirection stays overridable).
- Any embedded-skill assertion covering the entry point passes.
- The existing command-flow assertions in
  `internal/taskrail/shippable_skills_test.go` are migrated to the
  `${TASKRAIL:-taskrail}` form: they currently grep literal `taskrail <cmd>`
  substrings that this rewrite invalidates (`taskrail} repair` no longer
  contains `taskrail repair`). Rework each so it matches the resolved
  subcommand, not the hardcoded binary prefix.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass.
