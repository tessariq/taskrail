# T-099 coverage --gaps: structural gap signals (missing-verification, dependency-anomaly, under-decomposed)

## Description

Gap analysis, per `specs/v0.4.0.md#gap-analysis`, surfaces *likely missing work* as
advisory, report-only signals over the existing task graph and active spec. Build the
**structural (mechanical) half** as a `--gaps` mode on the existing read-only `coverage`
command (reuses its loaded state/tasks/spec parse). Signals must be mechanical and
inspectable — no semantic inference (`specs/v0.4.0.md#explicitly-excluded`). The
judgement-heavy semantic half is deferred to the `taskrail-gap` skill (T-101).

Boundary versus existing `coverage`: `coverage` answers "is this spec area linked to
any task?"; `--gaps` answers "does a covered area's task cluster lack a structural
companion?".

## Acceptance

- `coverage --gaps` emits, per covered active-spec area, these mechanical candidates:
  1. **missing-verification** — an area whose work is `done`/`completed` with no
     recorded verification (`last_verification_result` empty / no verify artifact);
  2. **dependency-anomaly** — advisory graph signals not already fatal under `validate`
     (e.g. a `blocked` task with no task that would unblock it; a task isolated from an
     otherwise-linked area's cluster);
  3. **under-decomposed-area** — an area whose `Requirements:` bullet count far exceeds
     its linked task count (count-based heuristic, flagged advisory).
- Strictly read-only: never writes `planning/STATE.md` or task files; no post-run
  `git status` follow-up is needed (mirrors `coverage`/`status`/`stats`).
- Every signal is framed as an advisory candidate ("promote to a task if real"), never
  auto-created state.
- `--json` mirrors the report for agent consumption.
- Add a matching "Notes On Repository Behavior" entry in `AGENTS.md` recording that
  `coverage --gaps` stays read-only and advisory.
