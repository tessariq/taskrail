# T-060 Add taskrail spec activate <version> to advance active spec (CLI-only STATE writer)

## Description

`STATE.md` declares `active_spec_version` and `active_spec_path`, but no command
advances them and hand-editing `STATE.md` is forbidden, so moving from one spec
version to the next (for example `v0.2.0` -> `v0.3.0`) has no sanctioned path.
Add `taskrail spec activate <version>` (a `spec` parent command with an
`activate` subcommand): validate that the target spec file exists (for example
`specs/<version>.md`) and follows the versioned-specs convention; set
`active_spec_version` and `active_spec_path` in `STATE.md` — the CLI stays the
only writer of that state; re-render `STATE.md` and re-run validation, reporting
the result; support `--json`. Activation only repoints the active spec; it never
advances task status or fabricates work. Spec:
`specs/v0.3.0.md#active-spec-activation`.

## Acceptance

- `taskrail spec activate <version>` sets `active_spec_version` and
  `active_spec_path` in `STATE.md`, re-renders it, and re-runs validation.
- A non-existent or non-conforming target version is rejected with a clear
  error and no write.
- Activation never changes task status fields or task files.
- `taskrail spec activate <version> --json` emits a machine-readable result.
- `README.md` is updated for the new command; the `CHANGELOG.md` entry folds
  into the single `spec` commands entry shared across T-060/T-062/T-063 (one
  entry per user-facing feature, not one per command).
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass.

## Verification Notes

- In a sandbox, activate a valid version and confirm the regenerated `STATE.md`
  and passing validation; attempt an invalid version and confirm the error and
  that `STATE.md` is unchanged.
