# T-060 Add taskrail spec activate <version> to advance active spec (CLI-only STATE writer)

## Description

`STATE.md` declares `active_spec_version` and `active_spec_path`, but no command
advances them and hand-editing `STATE.md` is forbidden, so moving from one spec
version to the next (for example `v0.2.0` -> `v0.3.0`) has no sanctioned path.
Add `taskrail spec activate <version>` (a `spec` parent command with an
`activate` subcommand): validate that the target spec file exists (for example
`specs/<version>.md`) and follows the versioned-specs convention; set
`active_spec_version` and `active_spec_path` in `STATE.md` — the CLI stays the
only writer of that state; re-render `STATE.md` and re-run validation, reporting
the result; support `--json`. Activation only repoints the active spec; it never
advances task status or fabricates work. Landing this command also retires the
one-time bootstrap `continuation_note` in `STATE.md` (which recorded the hand
edit that advanced v0.2.0 -> v0.3.0 because no `spec activate` existed yet): the
sanctioned path now exists, so the caveat is obsolete and must not linger in
committed state. Spec: `specs/v0.3.0.md#active-spec-activation`.

## Acceptance

- `taskrail spec activate <version>` sets `active_spec_version` and
  `active_spec_path` in `STATE.md`, re-renders it, and re-runs validation.
- A non-existent or non-conforming target version is rejected with a clear
  error and no write.
- Activation never changes task status fields or task files.
- The stale one-time bootstrap `continuation_note` in `STATE.md` is removed as
  part of this task; committed `STATE.md` carries no hand-edit caveat once the
  sanctioned activation path exists. The removal is a one-time sanctioned edit in
  this task's commit — symmetric with how the note was added (the note itself
  documents that it pierces the CLI-only-writer invariant exactly once). Do NOT
  add note-clearing logic to `spec activate`: the command must not special-case a
  particular note string, and a free-text note is not a deterministic `repair`
  case. `spec activate` re-renders `STATE.md` around the already-removed note.
- `AGENTS.md` is updated to record `spec activate` as a new `STATE.md` writer:
  add it to the "Notes On Repository Behavior" writer enumeration (alongside
  `start`/`next`/`verify`/`complete`/`block`) and to the Change Checklist's
  run-`git status`-and-stage-regenerated-`STATE.md` item, so the agent contract
  stays correct.
- `taskrail spec activate <version> --json` emits a machine-readable result.
- `README.md` is updated for the new command; the `CHANGELOG.md` entry folds
  into the single `spec` commands entry shared across T-060/T-062/T-063 (one
  entry per user-facing feature, not one per command). Create that shared entry
  if it is absent and append to it if present, so these tasks landing in any
  order never produce a duplicate line.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass.

## Verification Notes

- In a sandbox, activate a valid version and confirm the regenerated `STATE.md`
  and passing validation; attempt an invalid version and confirm the error and
  that `STATE.md` is unchanged.

## Implementation Notes

- Continuation-note removal mechanism (decided): a one-time sanctioned hand-edit
  of `STATE.md` in this task's commit removes the bootstrap `continuation_note`.
  This is the symmetric counterpart to the sanctioned pierce that added it; it
  adds no command logic. `spec activate` stays a generic repoint-only writer.
