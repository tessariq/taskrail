# T-037 Migrate the developer toolchain to mise

## Description

Taskrail has no pinned developer toolchain. Contributors install Go, `task`, and
`lefthook` by hand, and nothing provisions them on a fresh clone — the opt-in
lefthook hooks only activate after a manual `go install ...lefthook` plus
`task hooks:install`. Adopt [mise](https://mise.jdx.dev) as the optional,
documented way to pin and provision the developer environment, matching sibling
repositories.

## Acceptance

- A committed `mise.toml` pins the toolchain to explicit versions: Go (matching
  `go.mod`, `1.26`), `task` (go-task), and `lefthook` (the version referenced by
  `Taskfile.yml` `hooks:install` and `README.md`).
- `mise install` provisions the full toolchain on a fresh clone.
- A mise task (for example `mise run setup`) provisions the toolchain and wires
  the opt-in git hooks by running `lefthook install`, so one command takes a
  clean checkout to hooks-installed.
- mise stays optional convenience: direct `go` commands remain canonical and the
  existing `Taskfile.yml` targets keep working without mise.
- Pinned versions are a single source of truth: the `lefthook` pin in `mise.toml`
  matches the `task hooks:install` guidance, and the Go pin matches `go.mod`.
- Docs updated: `AGENTS.md` (Toolchain And Environment) and `README.md`
  (Development) describe `mise install` / `mise run setup` and note mise is
  optional.
- Optional: CI (`.github/workflows/ci.yml`) adopts mise to source the same pinned
  versions; if deferred, record why in the task notes.
- `go run ./cmd/taskrail validate` stays green and `./scripts/check-skill-mirrors.sh`
  is unaffected.

## Verification Notes

- Sandbox: in a fresh clone (or temp copy), run `mise install` then
  `mise run setup`; confirm `go`, `task`, and `lefthook` resolve at the pinned
  versions and that `.git/hooks` are lefthook-managed.
- Run `task build` and `go test ./...` under the mise-provisioned toolchain.
- Persist manual-test artifacts under `planning/artifacts/manual-test/T-037/`.

## Implementation Notes

- Pin `lefthook` to the version already referenced in `Taskfile.yml`
  (`hooks:install`) and `README.md` so there is one source of truth.
- Keep `task` as the command runner; mise can also provide the `task` binary.
- Do not make mise mandatory — taskrail keeps `task` and `go` usable directly.
- Consider `jdx/mise-action` for CI if adopting mise there.
- CI mise adoption deferred (kept optional per acceptance): `.github/workflows/ci.yml`
  already sources Go from `go.mod` via `actions/setup-go` (`go-version-file`), so it
  needs no mise-provided Go, and CI runs neither `task` nor `lefthook`. Adding mise
  there would introduce a second toolchain path for no coverage gain. The
  single-source-of-truth invariant is instead enforced by the `internal/toolchain`
  guard test, which CI already runs via `go test ./...`. Revisit if CI starts
  needing `task`/`lefthook`.
- Pin drift is guarded by `internal/toolchain/versions_test.go`: `go` pin ==
  `go.mod`, `lefthook` pin == every `lefthook@vX.Y.Z` in `Taskfile.yml`/`README.md`.
