# T-080 Add implementation-coverage dimension to coverage/status/stats (report-only)

## Description

`taskrail coverage` (T-059) reports the **decomposition** figure — a coverable
spec area counts covered when it has at least one linked, non-cancelled task,
regardless of status. That answers "is the work planned?" but not "can we
release it?". Add the second dimension the amended spec defines
(`specs/v0.3.0.md`, *Coverage Checks*): **implementation coverage** — an area is
covered only when *every* non-cancelled task linked to it, including through the
roll-up rule (a `###` area's `####` children), is `completed`. Compute it in the
same shared coverage capability T-059 exposes (no re-slugging, no duplicated
linkage logic) so both figures share one denominator (the coverable-area set)
and one anchor resolution. Surface both figures everywhere the decomposition
figure already appears: `taskrail coverage` (human + `--json`), the `status`
one-line coverage summary (T-056), and the `stats` per-area coverage table
(T-057). Implementation coverage is **report-only**: it never gates. `coverage
--min` (T-066) and this repo's push CI (T-079) stay bound to the decomposition
figure — during development most areas are decomposed but not yet implemented,
so gating a push on implementation coverage would red every run until release.
The implementation figure instead informs the pre-release check (T-061). Spec:
`specs/v0.3.0.md#deterministic-linkage-analysis`.

## Acceptance

- The shared coverage capability computes both a decomposition percentage
  (≥1 non-cancelled linked task) and an implementation percentage (all
  non-cancelled linked tasks `completed`, roll-up respected) over the same
  coverable-area set; per-area breakdown distinguishes the two states
  (uncovered / decomposed-not-implemented / implemented).
- `taskrail coverage` (human + `--json`) reports both figures and the per-area
  state; `status` (T-056) shows both in its one-line coverage summary and
  `stats` (T-057) shows both in its per-area table.
- Implementation coverage is report-only: `coverage --min` (T-066) and the T-079
  CI gate remain bound to the decomposition figure; a low implementation figure
  never makes `coverage --min`, `status`, `stats`, or `validate` exit non-zero.
- The `N/A` no-coverable-areas rule (T-059) applies to both figures: an
  unscoreable spec reports `N/A` (JSON `null`) for decomposition and
  implementation alike, never a false 0%/100%.
- On the dogfood spec `specs/v0.3.0.md`, decomposition reports 100% and
  implementation reports the true completed-task ratio (well under 100% while
  v0.3.0 is in progress) with no false gap under the coverable-area convention.
- Tests cover the implementation-percentage math (including roll-up and a
  partially-completed area reading decomposed-not-implemented) and the
  report-only guarantee (a low implementation figure does not gate).
- `README.md` and `CHANGELOG.md` are updated: the two-dimension coverage output
  folds into the existing `coverage` entry rather than adding a separate line.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass.

## Verification Notes

- In a sandbox with a known spec/task set where one area's tasks are all
  completed, one area has a mix of completed and todo tasks, and one area is
  uncovered, run `coverage`, `status`, and `stats` (human + `--json`); assert
  the decomposition and implementation figures, the per-area states, and that a
  low implementation figure gates nothing (`coverage --min` at the decomposition
  value still exits zero).
