# T-026 Strengthen spec and task linkage validation

## Description

`taskrail validate` already checks that each task has a `spec_ref`, that the
referenced path and heading anchor resolve (`validateSpecRef` /
`collectHeadingAnchors` in `internal/taskrail/service.go`), and that
dependencies point to existing task IDs. The spec's "Better Validation Around
Spec And Task Linkage" asks for stronger checks, and the "Portable Committed
State" section notes that today `validate` does not flag dangling artifact
references, so the rot is invisible.

Add three checks:

- flag any committed/tracked file that references a gitignored
  `planning/artifacts/` path (makes the rot from T-021/T-022 visible),
- detect dependency cycles among tasks,
- detect duplicate task IDs.

This depends on T-021/T-022/T-023 so the writers have stopped producing
artifact references and the existing dangling references are already scrubbed;
otherwise the new check would immediately fail on self-produced content.

## Acceptance

- `validate` flags committed references into gitignored `planning/artifacts/`
  content (in `STATE.md` frontmatter and task files), naming the offending file
  and field.
- `validate` detects dependency cycles among tasks and reports them.
- `validate` detects duplicate task IDs and reports them.
- Existing checks (spec_ref present, anchor resolves, dependencies exist) are
  unchanged.
- After the T-023 scrub, `validate` reports valid on this repository.
- Unit tests using temporary directories cover a dangling-reference violation, a
  cycle violation, a duplicate-id violation, and a clean pass.

## Verification Notes

- Run `go test ./internal/taskrail`.
- Run `go run ./cmd/taskrail validate` on this repository and confirm it reports
  valid after the scrub.

## Implementation Notes

- The validate path and helpers are in `internal/taskrail/service.go`
  (`validateSpecRef`, `dependenciesResolved`, `collectHeadingAnchors`).
- Detect gitignored references by matching the known gitignored
  `planning/artifacts/` prefix in committed `STATE.md` and task bodies/frontmatter;
  pattern-match the prefix rather than shelling out to git.
- Order after T-021/T-022/T-023 so the new check passes on a clean tree.
