# T-017 Write extracted release notes to a runner-temp path

## Description

The release workflow extracts CHANGELOG notes into `release-notes.md` at the
repository root. The first `v0.1.0` release attempt failed because GoReleaser's
clean-tree check flagged that untracked file (`git is in a dirty state`). The
immediate fix gitignored `release-notes.md`, and a non-empty guard now blocks a
blank body. As belt-and-suspenders, write the notes outside the working tree
entirely so the repo can never be made dirty by the release step.

## Acceptance

- The release workflow writes extracted notes to a path under `${RUNNER_TEMP}`
  (or another location outside the checked-out tree), not the repository root.
- GoReleaser is invoked with `--release-notes` pointing at that path.
- `release-notes.md` no longer needs a `.gitignore` entry to keep the tree clean
  (the entry may be removed once the file is gone from the tree).
- A tag push still produces a published GitHub Release with a non-empty body.

## Verification Notes

- Trigger a release (or a dry run) and confirm GoReleaser's clean-tree check
  passes with no `.gitignore` workaround for the notes file.
- Confirm the published release body matches the CHANGELOG section.

## Implementation Notes

- Note: GoReleaser action `args` do not reliably expand shell env vars, so the
  temp path likely needs to be passed via a step-level construction (for
  example writing the path into `$GITHUB_ENV` or using the action's `workdir`
  semantics) rather than embedding `${RUNNER_TEMP}` directly in `args`.
- Discovered during the `v0.1.0` release (run 28017787715 failure).
