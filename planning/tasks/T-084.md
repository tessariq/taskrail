# T-084 Guard docs skill lists against the shippable/dogfooding sets

## Description

Follow-up from T-054. The prose skill lists in `docs/workflow/skills-overview.md`
and `docs/workflow/skills-productization.md` (Shippable vs Dogfooding-only) are
hand-maintained and enforced by nothing. During T-054 they silently drifted:
`skills-overview.md` still listed `autonomous-recovery` as dogfooding-only —
contradicting the code and the productization doc — until a reviewer caught it.
Nothing fails when these docs disagree with the authoritative sets in
`internal/taskrail/shippable_skills_test.go` (`shippableSkills` /
`dogfoodingOnlySkills`), so the next skill graduation can regress them again.

Add a test that asserts these docs agree with the authoritative Go lists, so a
skill moving between sets fails CI unless both docs are updated. Keep it a plain
package test parsing the markdown; do not add a doc-generation runtime or a new
product command (out of scope, and the docs stay human-authored).

Spec: `specs/v0.3.0.md#ship-autonomous-recovery-and-autonomous-manual-test`.

## Acceptance

- A test in `internal/taskrail` fails when a skill in `shippableSkills` is absent
  from (or a `dogfoodingOnlySkills` skill is present in) the Shippable/
  Dogfooding-only lists of `docs/workflow/skills-overview.md` and
  `docs/workflow/skills-productization.md`, and passes when both docs agree.
- Demonstrate the guard bites: a deliberate mismatch (move one skill in a doc
  list without changing the Go list) makes the new test fail; reverting restores
  green.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass.

## Verification Notes

- Record the failing-then-passing test output demonstrating the guard bites.

## Implementation Notes

- Product-only onboarding skills (`taskrail-import`, `taskrail-retrofit`) live in
  `shippableSkills` but have no dogfooding counterpart; the doc parser must treat
  them as expected-shippable, not flag them as missing dogfooding entries.
