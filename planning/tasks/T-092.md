# T-092 Reconcile status_summary on complete when other tasks remain blocked

## Description

Surfaced in T-087's go-reviewer pass. `finishTask`'s non-blocked (complete)
branch (`internal/taskrail/transitions.go:467`) unconditionally resets
`status_summary = "idle"` / `next_action = "Select the next eligible task"` once
`current_task == ""`, regardless of whether other tasks remain blocked in the
ledger. This is asymmetric with `Unblock`
(`internal/taskrail/transitions.go:164`), which checks `len(remaining) > 0` and
stays `"blocked"`, pointing `next_action` at the last remaining blocker.

Repro: block T-003 while idle (`status_summary` → `blocked`), then start and
complete an unrelated T-004. `status_summary` flips to `idle` even though T-003
is still blocked and still recorded in the blockers ledger — a snapshot that
under-reports outstanding blockers, the same class of drift T-087 fixed for the
block direction.

Decide whether `complete` (and any other idle-reset path) should mirror
`Unblock`'s remaining-blocker reconciliation, or whether idle-on-complete is
intentional. If aligning, factor the shared reconciliation so `Unblock`,
`finishTask` block, and `finishTask` complete stay consistent.

Follow-up derived from T-087's verification or discovery.

## Acceptance

- Completing a task while another task remains blocked leaves `status_summary`
  reconciled consistently with `Unblock` (either both stay `blocked` pointing at
  the remaining blocker, or the divergence is documented as intentional with a
  rationale in code).
- Add a direct assertion for the baseline no-blockers-remain path: completing the
  active task with no other blockers sets `status_summary = "idle"` and
  `next_action = "Select the next eligible task"`. Today
  `TestStartCompleteAndBlockUpdateState` (`internal/taskrail/service_test.go:229`)
  only checks `current_task == ""` after `complete`, so the idle summary reset
  rests on the single-active-task invariant rather than a test — pin it directly.
- No regression to T-087: blocking a todo while another task is `in_progress`
  still preserves the active task's summary.
- Table-driven unit test covers both complete-while-others-blocked and the
  baseline complete-while-idle-clears-to-idle case. TDD: start with a failing
  test.
