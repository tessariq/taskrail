# T-049 v0.2.0 gap/drift — pre-release spec check

## Description

Pre-release gap / drift analysis for the **v0.2.0** release. Runs after the version's
implementation work lands and gates the release: v0.2.0 is not tagged until this check
passes. The spec is the source of truth: [specs/v0.2.0.md](../../specs/v0.2.0.md). Do not
duplicate spec content here.

## Release check

- Re-read the spec, focusing on **Goals** and **Potential Features**, plus the
  **Explicitly Excluded** and **Recommendation About LLM Support** sections (drift guards).
- Walk the codebase against each intended feature/goal. A **gap** is a spec commitment not
  met (missing or incomplete). A **drift** is built behavior that diverges from the spec's
  intent (implemented differently than specified, spec silently outgrown, or an
  Explicitly-Excluded item crept in).
- Run the repo's stated validation as evidence: `gofmt -l .`, `go vet ./...`,
  `go test ./...`, `go run ./cmd/taskrail validate`; note results.

## Findings and follow-ups

- Close with `taskrail verify T-049 --result pass|fail --summary "…"`, which writes an
  artifact under `planning/artifacts/verify/` and updates `STATE.last_verification_result`.
- For each gap or drift that should enter the backlog, add `--create-followup
  --followup-title "…" --followup-description "…" --followup-priority …` (or use the
  `autonomous-verify` skill). Use `--result pass` only when no unresolved gap or drift
  remains; otherwise the version is not ready to release.

## Acceptance

- Every v0.2.0 spec goal/feature walked against the codebase; each classified met / gap / drift.
- Stated validation (`gofmt -l .`, `go vet ./...`, `go test ./...`, `taskrail validate`) run and results recorded.
- Every gap/drift either resolved or captured as a follow-up task before closing.
- `--result pass` only when no unresolved gap or drift remains.
