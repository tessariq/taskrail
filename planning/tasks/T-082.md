# T-082 Make taskrail:check portable to native Windows (replace mktemp/cmp)

## Description

Follow-up from T-074. The `taskrail:check` Taskfile target (freshness guard for
the on-PATH `taskrail`) is written for a POSIX/CI shell: it uses `mktemp`, `cmp`,
and `trap`, plus `-o bin/taskrail` output naming. These external coreutils are
not present on a stock native Windows install (no Git-for-Windows/MSYS/WSL on
PATH), so `task taskrail:check` fails there even though `taskrail:install`
already produces a `.exe` and mise puts `./bin` on PATH.

Today this is inert for CI â€” the freshness check runs only in the `ubuntu-latest`
`checks` job â€” and the `desc` documents the POSIX/CI scope. This task makes the
check work for a contributor running `task taskrail:check` on native Windows,
without adding a runtime dependency or new product CLI behavior.

Spec: `specs/v0.3.0.md#goals` (build/CI hygiene, single-source toolchain).

## Acceptance

- `task taskrail:check` succeeds against a fresh build and fails loud on a stale
  binary on native Windows (PowerShell/cmd default shell), not only on
  POSIX shells.
- No reliance on `mktemp`/`cmp`/`trap` being on PATH; use a portable mechanism
  (e.g. Go/Task-native temp handling and a byte comparison that works cross-OS),
  or gate the target per-OS with an equivalent Windows path.
- The Windows binary name (`taskrail.exe`) is resolved correctly by the check
  (matches what `taskrail:install` writes).
- Keep the build byte-reproducible invariant intact
  (`CGO_ENABLED=0 -trimpath`), guarded by
  `TestTaskrailBuildsShareReproducibleFlags`.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass;
  `internal/toolchain` guards still pass.

## Verification Notes
