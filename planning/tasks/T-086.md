# T-086 Add taskrail unblock to reverse blocked back to todo

## Description

`block` moves a task to `blocked` and records its reason
(`internal/taskrail/transitions.go:122`), but nothing reverses it. `Start`
accepts only a `todo` task (`transitions.go:90`), and `repair` only ever
rewrites `STATE.md`, never a task file, so it cannot advance a status
(`repair.go:111`). A blocked task is therefore a CLI dead end whose only escape
is hand-editing task frontmatter — which `AGENTS.md` forbids once the CLI
exists. Add the missing inverse: `taskrail unblock <task-id>`.

See spec `specs/v0.3.0.md#task-unblocking`.

## Acceptance

- `taskrail unblock <task-id>` returns a `blocked` task to `todo`, rewriting the
  task file's `status` and `updated_at`, so it re-enters `next` selection.
- On any non-`blocked` status the command errors and writes nothing — no task
  file and no `STATE.md` change (mirror `start`'s `todo`-only guard at
  `transitions.go:90`).
- The task's entry is dropped from the `STATE.md` `blockers` list while the
  reasons of other still-blocked tasks are preserved (mirror the drop-only logic
  in `finishTask`, `transitions.go:388`).
- An optional `--reason` flag (paired with `block --reason`) appends a
  timestamped entry to the task's Implementation Notes via `appendTaskNote`
  (`transitions.go:378`) when non-empty; it is never re-added to the `STATE.md`
  `blockers` list. Omitting it unblocks with no note (like `block`/`complete`
  with an empty flag).
- `STATE.md` is re-rendered and validation re-run after the transition; the
  command reports the result and supports `--json` (match the `block`/`start`
  command shape and `jsonOption` wiring).
- The command is registered in `cmd/taskrail/root.go` and documented in
  `README.md` and `CHANGELOG.md` (Unreleased); a one-to-two-line changelog entry
  leading with `taskrail unblock`.
- Table-driven unit tests cover: blocked -> todo transition, the non-blocked
  guard (todo/in_progress/completed rejected, no write), blocker-entry removal
  with a second blocked task's reason retained, `--reason` appending a task note
  (and empty `--reason` appending none), and `--json` output. TDD: start each
  with a failing test.
