# T-038 Split internal/taskrail/service.go into focused modules

## Description

`internal/taskrail/service.go` has grown to ~1270 lines and 53 functions, well
past the 800-line hard ceiling in AGENTS.md (which already flagged the file as a
split candidate back when it was ~640 lines). The file now mixes several
cohesive concerns that can move into their own files without any behavior
change: validation (`validate*` plus the T-026 linkage helpers), tracked-work
transitions (`start`/`complete`/`block`/`verify`/`next`), state and task
load/save IO, and init/retrofit. Splitting along these seams restores the
"many small, focused files" norm and makes the validation and transition logic
easier to inspect.

Note: the `spec_ref` is a loose fit. This is an internal maintainability
refactor with no matching product-spec heading; it is anchored to the linkage
validation section because that is the closest existing anchor. If a dedicated
"internal structure / maintainability" spec section is later added, re-point
this task's `spec_ref` at it.

## Acceptance

- `internal/taskrail/service.go` is reduced under the 800-line ceiling by
  extracting cohesive units into new files in the same package (e.g.
  `validation.go`, `transitions.go`, `state_io.go`).
- No exported API, behavior, or state/task schema changes; this is a pure
  code-organization refactor.
- `gofmt -l .` clean, `go vet ./...` clean, `go test ./...` passes unchanged.
- `go run ./cmd/taskrail validate` still reports valid on this repository.

## Verification Notes

- Run `go test ./...` and confirm the suite passes with no test changes beyond
  moving code.
- Run `go run ./cmd/taskrail validate` and confirm it reports valid.

## Implementation Notes

- Discovered during T-026 (linkage validation), which added ~200 lines to an
