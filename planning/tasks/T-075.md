# T-075 spec activate: orphan callout for tasks still pointing at the previous spec

## Description

After `taskrail spec activate <version>` (T-060) repoints the active spec, list
the tasks whose `spec_ref` still points at the *previously* active spec — they
become orphans the moment the active spec moves, which is exactly when the
migration debt is most actionable. Reuses the shared orphan computation from
`taskrail coverage` (T-059); no duplicated logic. The callout is informational:
activation still succeeds and exits zero regardless, and it never touches task
status or task files. Complements T-067 (which prints the coverage summary after
activation) by adding the orphan side of the migration picture. Spec:
`specs/v0.3.0.md#active-spec-activation`.

## Acceptance

- After a successful `spec activate <version>`, the command lists tasks whose
  `spec_ref` still points at the previously active spec (the new orphans),
  reusing the shared T-059 orphan capability (no duplicated computation).
- Coverage/orphan output is informational: activation exits zero regardless and
  never changes task status or task files.
- `--json` includes the orphan list alongside the activation result.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass.

## Verification Notes

- In a sandbox with a task still pointing at the old spec, activate a new
  version and confirm the orphan callout lists it, activation exits zero, and no
  task files change.
