# T-074 Harden CI and build scripts to route commands via Taskfile and Mise

## Description

Tighten CI and the local build so every build/test/lint/validate command is
invoked through a `Taskfile.yml` target rather than ad-hoc inline shell, and so
every tool is version-managed by mise (`mise.toml` the single source of tool
versions, CI provisioning via `jdx/mise-action`). This makes a local `task <x>`
run identical to CI and keeps the toolchain pins in one place, consistent with
the existing `internal/toolchain` guard. No new product behavior — this is
build/CI hygiene that the rest of the v0.3.0 work builds on. Infra work gating
the feature tasks. Spec: `specs/v0.3.0.md#goals`.

Also thread in the mechanism the adopted skills (T-051) rely on to resolve the
Taskrail binary in this repo **without an env override**: build the working-tree
`./cmd/taskrail` and expose it as `taskrail` on the **mise-provided PATH**, so a
plain `${TASKRAIL:-taskrail}` fallback resolves to the working tree in any shell
or runner (no `TASKRAIL` env, no Claude-specific config). `mise run setup` (and a
Taskfile target) build it and place it on the mise-managed path; the same PATH
contract already provides `go`/`task`/`lefthook`, so no new precondition is
added and a missing mise PATH fails loudly and totally rather than silently
resolving a stale `taskrail`. Add a freshness guard (rebuild on setup/pretest,
plus a Taskfile/CI check that the resolved `taskrail` matches the working tree)
so an out-of-date binary is caught, not silently used.

## Acceptance

- `.github/workflows/ci.yml` invokes build/test/lint/validate through Taskfile
  targets (`task ...`) instead of duplicating raw commands inline.
- All tools used in CI and locally are provisioned via mise (`mise.toml`),
  asserted as the single source of tool versions; no tool version is pinned in
  more than one place.
- A Taskfile target exists for each command CI runs, so `task <x>` locally
  matches CI.
- `mise run setup` (and a Taskfile target) build `./cmd/taskrail` and place it on
  the mise-provided PATH, so in a plain, non-activated shell bare `taskrail`
  resolves to the working-tree build. No `TASKRAIL` env override is set anywhere
  (the fallback used by the adopted skills, T-051, is correct by construction).
- A freshness guard (rebuild on setup/pretest plus a Taskfile/CI check comparing
  the resolved `taskrail` to the working tree) fails loud when the on-PATH
  `taskrail` is stale, so a skill never silently runs an out-of-date binary.
- The `internal/toolchain` guards still pass and are not bypassed.
- `gofmt -l .`, `go vet ./...`, `go test ./...`, and `taskrail validate` pass.

## Verification Notes

- Run each CI command locally via its Taskfile target under mise and confirm it
  matches CI behavior; confirm no tool version is declared outside `mise.toml`.
- After `mise run setup`, in a non-activated shell with `TASKRAIL` unset, confirm
  bare `taskrail` resolves to the freshly-built working-tree binary; make it stale
  and confirm the freshness guard fails.
