# T-083 Retain all blocked-task reasons in STATE.md blockers (finishTask overwrite)

## Description

`finishTask` (`internal/taskrail/transitions.go`) sets
`state.Frontmatter.Blockers = []string{fmt.Sprintf("%s: %s", taskID, note)}` on
each `block`, **overwriting** the slice instead of appending. So `STATE.md`'s
`blockers` list only ever retains the most recently blocked task's reason. When
two or more tasks are blocked, earlier reasons are lost from committed state.

Discovered while implementing `taskrail status` (T-056): status reads reasons
from that list, so an earlier blocked task has no recorded reason. T-056 mitigated
the *symptom* by rendering an explicit `UnrecordedBlockerReason` marker instead of
a silent empty string, but the underlying state-fidelity bug remains here.

Fix `finishTask` (and the `block`/complete/unblock transitions) so the blockers
list retains one entry per currently-blocked task, and is pruned when a task
leaves the blocked state. This is a committed-state behavior change, so treat it
as risky: add focused transition tests and verify `validate`/`repair` still agree
on the blockers projection.

## Acceptance

- Blocking a second task while a first is still blocked keeps **both** reasons in
  `STATE.md` `blockers` (one entry per blocked task, `"<id>: <reason>"`).
- Completing/unblocking a task removes only its own blocker entry.
- `taskrail status` shows each blocked task's real recorded reason (the
  `UnrecordedBlockerReason` marker becomes unnecessary for the multi-block case).
- `validate` and `repair` remain consistent with the blockers projection.
- Tests cover the multi-blocked and unblock-one-of-many cases.

## Verification Notes

- Reproduce: block two tasks, inspect `planning/STATE.md` `blockers` and
  `taskrail status --json`; assert both reasons are present.
