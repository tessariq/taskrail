# T-035 Add the guided retrofit bootstrap flow

## Description

Tie the pieces into the spec's Guided Retrofit Bootstrap Flow: take an existing
repository plus human notes and produce `specs/`, `planning/tasks/`, and an
initial `STATE.md`. Combine layout detection (T-020), import-to-planning
(T-033), and the agent-driven path (T-034) into a single guided flow that
proposes a mapping, defaults to a dry run, and applies only on explicit
confirmation. Detection proposes; it never clobbers existing files without
confirmation.

## Acceptance

- A guided retrofit entry point (for example `taskrail init --retrofit <notes>`
  or `taskrail retrofit`) runs: detect layout (T-020), import notes to a
  planning bootstrap (T-033/T-034), and propose `specs/` plus `planning/tasks/`
  plus `STATE.md`.
- It is a dry run by default, shows the proposed mapping/diff, and applies only
  on explicit confirmation; no existing file is overwritten without
  confirmation.
- After apply, a `.taskrail/config.yml` marker is written (T-018/T-019) and
  `validate` is re-run and reported.
- Unit tests cover a non-standard repository retrofitted end to end in a
  temporary directory, including the no-clobber guarantee.

## Verification Notes

- Run `go test ./internal/taskrail`.
- Manual test in a temp sandbox: build a non-standard repo plus `notes.md`, run
  the retrofit dry run then apply, confirm `specs/`, `planning/`, and `STATE.md`
  are produced and `validate` passes; persist `plan.md` and `report.md` under
  `planning/artifacts/manual-test/T-035/<timestamp>/`.

## Implementation Notes

- Compose existing surfaces rather than forking new logic: T-020 detection,
  T-033/T-034 import, and the T-018/T-019 marker plus dry-run/apply primitive.
- Reuse one confirmation mechanism (the dry-run/diff/apply shape introduced in
  T-019) so retrofit, init upgrade, and detection stay consistent.
