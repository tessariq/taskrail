# T-018 Add .taskrail/config.yml layout marker and config-aware path discovery

## Description

The repository layout is currently hardcoded in `DiscoverPaths`
(`internal/taskrail/paths.go`): it walks up to the nearest `.git` directory and
then assumes a fixed layout (`specs/`, `planning/`, `planning/STATE.md`,
`planning/artifacts/`). No file marks a repository as Taskrail-managed and no
layout version is recorded.

Introduce a single machine-owned marker file, `.taskrail/config.yml`, that acts
as the "this repo uses Taskrail" signal, carries an explicit `layout_version`,
and records where Taskrail's directories live. Make path discovery read it,
falling back to the current layout when it is absent so the change is purely
additive. Human-facing content (`specs/`, `planning/`) stays visible and
top-level; only the marker and machine config live under `.taskrail/`.

This is the foundational primitive for versioned upgrade (T-019) and retrofit
detection (T-020).

## Acceptance

- A `.taskrail/config.yml` schema exists with at least a `layout_version` field
  and the Taskrail directory locations Taskrail already uses.
- `DiscoverPaths` resolves paths from `.taskrail/config.yml` when present.
- When `.taskrail/config.yml` is absent, `DiscoverPaths` falls back to the
  current hardcoded v0.1.0 layout with identical behavior (no migration needed
  for existing repositories).
- The config is minimal: no configuration knobs that nothing reads yet;
  arbitrary path relocation is out of scope for this task.
- Unit tests cover both the marker-present and marker-absent discovery paths
  using temporary directories.

## Verification Notes

- Run `go test ./internal/taskrail` and confirm discovery tests for both paths
  pass.
- Run `go run ./cmd/taskrail validate` in a repo with no marker and confirm it
  behaves exactly as before.

## Implementation Notes

- `gopkg.in/yaml.v3` is already a direct dependency (used for frontmatter), so
  YAML config adds no new dependency.
- Keep the fallback layout identical to the constants currently in `paths.go`.
