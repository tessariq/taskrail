# T-019 Make taskrail init version-aware with non-destructive upgrade

## Description

`taskrail init` is currently a blind `ensureDir` / `writeFileIfMissing` pass
(`internal/taskrail/service.go` `Init`). It is idempotent for creation but has
no awareness of layout version and no upgrade path.

Building on the layout marker from T-018, make `init` version-aware:

- empty repository: fresh init writes the current layout and the marker,
- existing v0.1.0 layout with no marker: detect it, write a marker pinning the
  current layout, and change nothing else (non-destructive adoption),
- marker present with an older `layout_version`: migrate to the current layout,
  defaulting to a dry run that shows the diff before applying, then re-running
  validation.

This mirrors the conservative discipline required of `taskrail repair`:
migration must never destroy or rewrite human-authored content and must be
reversible in effect (dry-run first, explicit apply).

## Acceptance

- Running `init` on an empty repository writes the current layout plus a
  `.taskrail/config.yml` with the current `layout_version`.
- Running `init` on an existing v0.1.0 layout with no marker writes a marker
  pinning the current layout and makes no other changes.
- When the marker records an older `layout_version`, `init` performs a migration
  that defaults to a dry run, prints the diff, and only applies on explicit
  confirmation, then re-runs validation and reports the result.
- Migration never deletes or rewrites human-authored content under `specs/` or
  `planning/`.
- Unit tests cover empty-repo, legacy-adoption, and older-version-migration
  paths using temporary directories.

- Keep `writeFileIfMissing` semantics for content creation; add version
