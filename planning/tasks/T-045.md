# T-045 Include task filename in saveTask write-error message

## Description

`saveTask` in `internal/taskrail/state_io.go` wraps its `os.WriteFile` failure
with the static string `write task file: %w` (added in T-039). It does not name
which task file failed. The parallel load path already includes the filename:
`fmt.Errorf("read task %s: %w", entry.Name(), err)` (same file). When `saveAll`
iterates several tasks and one write fails, the returned error has no identity —
the caller cannot tell which file failed.

Flagged as a LOW finding by the T-039 go-reviewer pass. Deferred from T-039 to
keep that change to the minimal error-wrapping scope named in its acceptance.

Note: `spec_ref` is inherited from T-039 and is a loose fit — this is an
internal error-message consistency nit with no matching product-spec heading.

## Acceptance

- `saveTask` includes the task filename in its write-error message, consistent
  with the read path (e.g. `fmt.Errorf("write task file %s: %w",
  filepath.Base(task.Filename), err)`).
- No other behavior change; `gofmt -l .` clean, `go vet ./...` clean,
  `go test ./...` passes.
- A test asserts the failing task's filename appears in the wrapped error and
  the `%w` chain still reaches the underlying `os.WriteFile` error.

## Verification Notes

- Extend `internal/taskrail/state_io_test.go` (`TestSaveTaskWrapsWriteError`) to
  assert the filename is present in the error string.
- Run `go test ./internal/taskrail` and `go run ./cmd/taskrail validate`.
