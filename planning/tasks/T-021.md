# T-021 Record a portable verification result in committed state

## Description

`taskrail verify` writes references to gitignored artifacts into the committed
`STATE.md` frontmatter. In `internal/taskrail/service.go`:

- `state.Frontmatter.LastVerificationResult` is set to
  `"%s for %s (%s)"` including `relReport`, a path under
  `planning/artifacts/verify/...` which `.gitignore` excludes.
- `state.Frontmatter.RelevantArtifacts` is set to `[]string{relPlan, relReport,
  relReportMarkdown}`, all gitignored.

A teammate cloning the repo gets `STATE.md` pointing at timestamped files that
exist only on the producer's machine. Make committed state portable: record the
verification result without a path into gitignored content.

## Acceptance

- `last_verification_result` records a portable summary (at least task id,
  result, and timestamp) with no path into a gitignored artifact.
- `relevant_artifacts` lists only committed paths, or is empty; it never lists a
  gitignored artifact path.
- The verify command still writes the full local evidence files under
  `planning/artifacts/verify/...` (no change to local evidence generation).
- If a producer-local pointer to evidence is kept, it lives in a gitignored
  side-channel, not in committed `STATE.md`.
- Unit tests assert that no gitignored path is written into state frontmatter
  after a verify, using temporary directories.

## Verification Notes

- Run `go test ./internal/taskrail` and confirm the new state-portability tests
  pass.
- Manually run a verify in a temp sandbox and confirm `STATE.md` contains no
  `planning/artifacts/` path; persist artifacts under
  `planning/artifacts/manual-test/T-021/<timestamp>/`.

## Implementation Notes

- The fields are `LastVerificationResult` and `RelevantArtifacts` in
  `internal/taskrail/types.go`; a structured portable shape may require a schema
  adjustment.
- Pair with T-022 (task notes) since both originate in the same verify writer.
