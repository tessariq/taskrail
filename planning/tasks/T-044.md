# T-044 Adopt mise as the CI toolchain provisioner

## Description

T-037 adopted `mise.toml` as the optional local toolchain but deferred CI, leaving
`.github/workflows/ci.yml` on `actions/setup-go` with the Go version sourced from
`go.mod`. That is a second toolchain path: local dev provisions via mise while CI
provisions via a different mechanism. Per follow-up direction, mise should be the
single toolchain provisioner for both local and CI so the pinned versions in
`mise.toml` are the one source of truth everywhere.

Spec: `specs/v0.2.0.md#mise-toolchain-management` ("the pinned versions are the
single source of truth shared by local setup and CI").

## Acceptance

- Every job in `.github/workflows/ci.yml` provisions its toolchain via
  `jdx/mise-action` (pinned to a major version, consistent with the repo's action
  pinning style) instead of `actions/setup-go`.
- CI no longer references `actions/setup-go`; the Go version on CI comes from
  `mise.toml` (which already matches `go.mod` via the T-037 guard test), so there
  is no second place pinning Go for CI.
- The canonical `task` runner is used on CI where a matching Taskfile target
  exists (at minimum `task test`), dogfooding the same command path contributors
  use locally.
- Lint/build gates that have no clean Taskfile equivalent (`gofmt` check, `go vet`,
  `CGO_ENABLED=0 go build`) may stay as direct commands, but run under the
  mise-provisioned toolchain.
- A guard test asserts CI is mise-provisioned (references `jdx/mise-action`, does
  not reference `actions/setup-go`) so the invariant cannot silently regress.
- `go run ./cmd/taskrail validate`, `go test ./...`, and
  `./scripts/check-skill-mirrors.sh` stay green.
- Docs (`AGENTS.md`, `README.md`) reflect that mise provisions the toolchain on CI
  as well as locally.

## Verification Notes

- Guard test: `go test ./internal/toolchain/` proves CI references mise-action and
  not setup-go.
- Lint the workflow YAML for validity (parse with a YAML tool / `yamllint` if
  available, or a Go yaml round-trip).
- CI cannot be fully exercised locally; record that live validation happens on the
  next push/PR run and note the tradeoff of dropping `actions/setup-go` caching.

## Implementation Notes

- Pin `jdx/mise-action` to a major version (repo uses `actions/checkout@v7`,
  `actions/setup-go@v6`); current latest is `v4`.
- `mise-action` runs `mise install` from `mise.toml` and activates the tools on
  PATH for later steps, so existing `go`/`gofmt` commands keep working.
- Dropping `actions/setup-go` also drops its `cache: true` Go module/build cache;
  acceptable for a small repo (mise-action caches tool installs). Revisit with an
  explicit `actions/cache` step only if CI wall time becomes a problem.
- Review decision (go-reviewer, T-044): the lost Go cache is a ~15–30s/job
  performance regression, not a correctness one, and reintroducing a cache-only
  `actions/setup-go` would defeat single-provisioner intent. Accepted as a known
  tradeoff (see above), not fixed in this task.
- `internal/toolchain/ci_test.go` checks actual `uses:` step values (comments and
  inline `# ...` stripped), so a historical comment mentioning `actions/setup-go`
  cannot false-fail the guard and a commented-out mise step cannot false-pass it.
