# T-039 Wrap save-path write errors with context in state_io.go

## Description

`saveState` and `saveTask` in `internal/taskrail/state_io.go` return the bare
`os.WriteFile` error without wrapping it in context, unlike the rest of the
package (e.g. `loadState` wraps with `"read state file: %w"`, `ensureDir` with
`"create directory %s: %w"`). On a write failure the caller sees only
`open /path: <errno>` with no indication of which logical operation failed
(state save vs task save). The repo Go style requires
`fmt.Errorf("context: %w", err)` at every level.

Flagged by the T-038 go-reviewer pass as a pre-existing MEDIUM. It was left out
of T-038 deliberately: T-038 was a pure verbatim-move refactor, and adding
wrapping changes the observable error string (an error-semantics change) that
belongs in its own task.

Note: the `spec_ref` is inherited from T-038 and is a loose fit — this is an
internal error-handling nit with no matching product-spec heading. Re-point it
if a dedicated maintainability spec section is added later.

## Acceptance

- `saveState` and `saveTask` wrap their `os.WriteFile` failures with
  `fmt.Errorf("<what>: %w", err)`, naming the logical operation (e.g.
  `write state file`, `write task file`).
- No other behavior change; `gofmt -l .` clean, `go vet ./...` clean,
  `go test ./...` passes.
- A test asserts the wrapped context is present on a write failure (e.g. via an
  unwritable path) and that `errors.Is`/`errors.Unwrap` still reaches the
  underlying `os.WriteFile` error.

## Verification Notes

- Add or extend a focused unit test in `internal/taskrail` that forces a write
  failure and asserts the error message and `%w` chain.
- Run `go test ./internal/taskrail` and `go run ./cmd/taskrail validate`.
