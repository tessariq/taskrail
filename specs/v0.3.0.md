# Taskrail v0.3.0

## Summary

Taskrail `v0.3.0` advances two threads at once.

The first is the **strategic** thread: spec-task consistency, decomposition
quality, and planning drift. This is where Taskrail becomes significantly more
valuable than a deterministic task-state CLI.

The second is an **operational and distribution** thread: give humans and agents
a read-only view of tracked-work state (`status`), aggregate metrics (`stats`),
a wider mechanical `repair` surface, and a single, converged skill-distribution
story so this repository stops maintaining custom skills that duplicate the
packaged set.

Both threads keep Taskrail provider-independent and free of any LLM integration
in the core, consistent with `v0.1.0` and `v0.2.0`.

## Goals

- Improve alignment between intent and execution.
- Detect missing work and spec/task drift earlier.
- Keep planning grounded in the active spec.
- Give a fast, read-only answer to "what is the state right now?" (`status`).
- Give aggregate insight into the task graph (`stats`).
- Widen conservative, mechanical state repair so recovery leans on the CLI
  rather than human judgement wherever a fix is genuinely deterministic.
- Converge skill distribution: one packaged skill set, consumed everywhere
  (including this repository), with no bespoke mirror to keep in sync.
- Distribute a Windows build through WinGet, automated from the existing
  tag-triggered release pipeline.
- Round out a small `spec` command family (list, show, add, activate) and a
  skill so agents author tasks against real spec anchors.

## Potential Features

### Spec-To-Task Decomposition Support

Help derive draft tasks from spec documents. This may be a guided workflow, an
import mode, a prompt- or skill-assisted flow, and later an optional built-in AI
path. Output must remain reviewable draft tasks, never silently committed state.

### Coverage Checks

Check whether spec requirements are represented by tasks:

- Are there spec areas (live headings) with no linked tasks?
- Are core requirements uncovered?

Coverage produces a percentage and a per-area breakdown that both `status` (as a
one-line summary) and `stats` (as a detailed table) can surface.

### Orphan Task Checks

Check whether tasks are meaningfully grounded in current specs and goals:

- Are there tasks with no spec linkage?
- Are there outdated tasks detached from current intent?

### Drift Detection

Check for inconsistencies between current specs and the current task graph:

- Did the spec evolve without corresponding tasks?
- Did tasks evolve beyond or away from the spec?

Keep this mechanical and inspectable. Deep semantic drift detection stays
excluded (see Explicitly Excluded).

### Gap Analysis

Help identify likely missing work: tests, verification tasks, docs, migration,
rollout, edge cases, cleanup.

### Deterministic Linkage Analysis

Of the strategic sections above, `v0.3.0` implements the deterministic subset —
Coverage Checks, Orphan Task Checks, and Drift Detection — as one report-only
command, `taskrail coverage`. Spec-To-Task Decomposition and Gap Analysis stay
exploratory: they need judgement and heuristics that risk the agent-framework
and semantic-analysis creep the Caution and Explicitly Excluded sections forbid,
so they are deferred to a later version.

`taskrail coverage` is strictly read-only — it never writes `STATE.md` or task
files — and reports, against the active spec:

- **Coverage** — live headings in the active spec with no linked task, plus a
  coverage percentage. This is the figure `status` and `stats` surface.
- **Orphans** — tasks whose `spec_ref` points outside the active spec, or is
  otherwise detached from current intent. (A `spec_ref` whose anchor does not
  exist at all stays a hard `validate` failure, not a soft coverage signal.)
- **Drift** — the two-directional summary: active-spec areas that gained no
  tasks, and tasks that point away from the active spec.

These are **advisory** signals, not `validate` violations: a coverage gap must
not make `validate` fail. `--json` mirrors the report for agent consumption. The
coverage computation is a shared internal capability that `stats` reuses for its
spec-coverage metric.

### Status Reporting

Add `taskrail status`: a fast, **strictly read-only** overview of current
tracked-work state.

Requirements:

- Never writes `planning/STATE.md` or task files. Unlike `next`, which dirties
  the working tree by persisting `next_action`/`updated_at` even on a selection
  probe, `status` must leave the tree untouched. It may compute the next
  eligible task using the same selection logic, but must not persist it.
- Default (human) view reports:
  - active spec version and path,
  - task counts by status (done / active / blocked / todo),
  - the next eligible task, explicitly marked as *not persisted*,
  - blocked tasks with their recorded reasons,
  - the last verification result (the path-free summary already kept in
    `STATE.md`),
  - a one-line coverage summary when Coverage Checks are available.
- `--json` emits the same information in a machine-readable, agent-consumable
  shape.

Rationale: agents and humans repeatedly need the current picture without the
side effect of `next` rewriting state. `validate` answers "is it consistent?"
and `next` answers "what do I do next (and record it)?"; neither answers "what
is the state right now?" without a write. `status` fills that gap safely.

Boundary against `stats`: `status` reports the *current operational snapshot*
(what is true now, what is next). `stats` reports *aggregate metrics across the
task graph*. They may share computed inputs (for example coverage) but serve
different questions.

### Statistics Reporting

Add `taskrail stats`: detailed aggregate statistics over the task graph.

Data model: **snapshot-only**, computed from the current task files and
`STATE.md`. Taskrail keeps no event log, and verification artifacts are
gitignored and ephemeral, so `stats` reports the current distribution, not
historical trends. Time-series/throughput analysis is explicitly out of scope
for `v0.3.0` (see Explicitly Excluded) to avoid introducing a new persistent
event-log surface.

Suggested metrics:

- counts and percentages by status,
- blocked ratio and the count of tasks with recorded blockers,
- spec-coverage percentage and a per-spec-area breakdown,
- dependency shape (for example tasks with unmet dependencies, longest
  dependency chain),
- last-verification tally where determinable from current state.

Like `status`, `stats` is read-only and supports `--json` for agent
consumption.

Rejected alternatives (deferred, not chosen for `v0.3.0`):

- Git-history-derived trends (walking `STATE.md`/task-file history): possible
  without new persistence but coupled to commit granularity; revisit later.
- A committed append-only event log: the most capable, but adds a new
  persistent product surface and schema; excluded here.

### Extended State Repair

`v0.2.0` shipped `taskrail repair` for a narrow mechanical set: a `current_task`
pointer disagreeing with the in_progress task, a stale `current_task_title`, and
stale rendered task counts. `v0.3.0` should audit tracked-work inconsistencies
and widen repair to cover every additional case that is **genuinely
deterministic** — where exactly one safe correction exists and applying it
neither advances a status nor fabricates work.

Candidate additional mechanical cases to evaluate (each only if a single safe
fix is provable):

- a stale focus/summary field derived from task files,
- other rendered `STATE.md` body fields that are pure projections of task state,
- dangling references that can be *reported* precisely (repair surfaces them;
  it does not invent the missing target).

Hard boundary — these stay human-resolved by design, not as a coverage gap:

- a missing `spec_ref` (repair cannot invent a spec linkage),
- a dependency cycle (repair cannot choose which edge to cut),
- more than one in_progress task (repair cannot know which is authoritative).

Repair must continue to only ever rewrite `STATE.md` — never a task file — so it
cannot advance a status or fabricate work. It defaults to a dry run showing the
diff and re-runs validation after `--apply`.

Non-goal: turning `repair` into a general fixer that resolves ambiguous state by
guessing. The invariant "never advance status or fabricate work" is absolute.

### Active Spec Activation

`STATE.md` declares `active_spec_version` and `active_spec_path`, but no command
advances them and hand-editing `STATE.md` is forbidden. Moving from one spec
version to the next (for example `v0.2.0` -> `v0.3.0`) therefore has no
sanctioned path today.

Add `taskrail spec activate <version>`:

- validates that the target spec file exists (for example `specs/<version>.md`)
  and follows the versioned-specs convention,
- sets `active_spec_version` and `active_spec_path` in `STATE.md` — the CLI
  stays the only writer of that state,
- re-renders `STATE.md` and re-runs validation, reporting the result,
- supports `--json`.

This keeps `STATE.md` authoritative and removes the last reason to hand-edit it.
Activation only repoints the active spec; it never advances task status or
fabricates work.

### Spec Inspection And Authoring Commands

Beyond `spec activate`, `v0.3.0` rounds out a small `spec` command family so
task authors and coding agents can discover and manage specs through the CLI
instead of reading and slugifying headings by hand.

- **`spec list`** — list the versioned specs under `specs/` and mark the active
  one. Read-only.
- **`spec show <version>`** — print a spec; with `--anchors`, list its live
  heading anchors exactly as `spec_ref` validation computes them, so
  `taskrail task new --spec-ref <path>#<anchor>` can be authored against real
  anchors without reverse-engineering the slug rule. Read-only.
- **`spec add <version>`** — scaffold `specs/<version>.md` with the standard
  section skeleton and update the `specs/README.md` reading order. This is the
  one writer in the group; it creates a spec file but never touches `STATE.md`
  or task state.

The read commands support `--json`.

#### Spec Command Skill

Ship a repo-agnostic skill (packaged, opt-in via `taskrail init --with-skills`,
invoking `${TASKRAIL:-taskrail}`) that guides coding agents through the spec
family: discover anchors with `spec show --anchors` before authoring a task,
inspect available specs with `spec list`, advance the active spec with `spec
activate`, and scaffold a new spec with `spec add`. It stays static,
provider-agnostic skill text consistent with the skills productization contract;
Taskrail still does not execute or orchestrate skills.

### Skill Distribution Convergence

`v0.2.0` shipped skills embedded in the binary, installed on opt-in via
`taskrail init --with-skills`. This repository still maintains a separate
canonical `skills/` set mirrored three ways (`skills/`, `.claude/skills/`,
`.agents/skills/`) and guarded by `scripts/check-skill-mirrors.sh`. The two
sets have drifted deliberately: dogfooding skills call `go run ./cmd/taskrail`,
shipped skills call the installed `taskrail`, and two dogfooding skills
(`autonomous-recovery`, `autonomous-manual-test`) are not shipped at all.

`v0.3.0` converges these into a single packaged skill set consumed everywhere.

#### Configurable Skill Entry Point

Shipped skill text must call the Taskrail binary through an overridable
indirection, `${TASKRAIL:-taskrail}`, instead of a hardcoded `taskrail` or
`go run ./cmd/taskrail`:

- Adopting repositories need nothing: the default resolves to the installed
  `taskrail` binary.
- This repository sets `TASKRAIL="go run ./cmd/taskrail"` in `mise.toml`
  `[env]` (committed), so any agent shell running under mise dogfoods the
  working tree, not a stale installed binary.

This removes the reason the dogfooding and shipped skills diverged.

#### This Repository Adopts The Packaged Skills

This repository stops maintaining a bespoke `skills/` source. It consumes the
packaged skills — the embedded set materialized by `taskrail init
--with-skills` — like any adopter.

- The custom `skills/` directory and `scripts/check-skill-mirrors.sh`
  three-way diff are retired.
- The materialized `.claude/skills/` and `.agents/skills/` copies stay
  **committed** (zero-setup clone). A single parity check replaces the old
  three-way mirror check: assert the committed copies equal the embedded
  `--with-skills` output. CI keeps them from drifting from the package.
- `AGENTS.md`, `docs/workflow/skills-productization.md`, and
  `docs/workflow/skills-overview.md` update to describe one packaged set with a
  configurable entry point, replacing the dogfooding-vs-shipped split.

#### Ship autonomous-recovery And autonomous-manual-test

Both dogfooding-only skills join the shipped set so the packaged set is a
superset this repository can adopt without losing capability:

- `autonomous-recovery` already routes every correction through `taskrail
  repair` and already forbids hand-editing authoritative state; the "falls back
  to manual edits" premise in the productization doc is stale. Shipping it needs
  only the entry-point change and the Extended State Repair work above, which
  shrinks its human-resolved residue. The productization doc's disposition table
  updates from "dogfooding-only" to shippable.
- `autonomous-manual-test` ships as a repo-agnostic skill, but its artifacts
  stay ephemeral and gitignored. Manual-test is **not** promoted to a product
  invariant: `init` does not provision `planning/artifacts/manual-test/` and
  `validate` remains unaware of it. This resolves the `v0.2.0` "Artifact And
  Init Consistency" deferral in favor of shipping the skill without a new
  invariant. Shipped skills continue to reference manual testing generically;
  the dedicated skill is available for agents that want the structured flow.

#### Force-Reinstall For Skill Upgrades

`taskrail init --with-skills` currently skips when skills are already installed,
so upgrading the binary never refreshes materialized skill files. Add a
`--force` that reinstalls the embedded skills over existing copies:

- Any locally-modified skill file is backed up (for example to a `.bak` or
  timestamped copy) before being overwritten, so edits are recoverable.
- Output reports what was overwritten and what was backed up.
- Without `--force`, behavior is unchanged: already-installed skills are left
  in place.

This gives adopters a clear upgrade path when a new Taskrail version ships
updated skills, and it is what this repository's parity check relies on to keep
committed copies aligned with the package.

### Windows Distribution via WinGet

`v0.3.0` should ship a Windows build and distribute it through the Windows
Package Manager (WinGet), automated from the existing tag-triggered GoReleaser
release pipeline (`.github/workflows/release.yml`, `.goreleaser.yaml`). Today
the pipeline builds only darwin and linux archives.

#### Distribution Mechanism

Use GoReleaser's built-in `winget` publisher — not the standalone
`winget-releaser` action. Taskrail already releases through GoReleaser, so the
`winget` block keeps distribution in one tool and one config, and its
open-source tier covers everything needed here (only MSI/NSIS output and PR
checkbox automation are Pro, and neither is required).

On each tagged release, GoReleaser generates the WinGet manifest set and opens a
pull request against `microsoft/winget-pkgs`.

#### Portable Single-Binary Packaging

Taskrail is a single binary, so no MSI or installer authoring is required.
Package it as `InstallerType: zip` with `NestedInstallerType: portable` (WinGet
supports zip archives since 1.5 and portable installers since 1.3). The Windows
release artifact is a `.zip` containing `taskrail.exe`.

#### Pipeline Requirements

- Add a Windows build target to `.goreleaser.yaml` (`windows/amd64` required;
  `windows/arm64` optional) producing a `.zip` archive.
- Add a `winget` block: `package_identifier: Tessariq.Taskrail`, publisher
  `Tessariq`, license, short description, and `publisher_url`.
- Point the block's `repository` at a fork of `microsoft/winget-pkgs` owned by
  the project's account/org, with `pull_request.enabled: true` and
  `pull_request.base` set to `microsoft/winget-pkgs`.
- Provide a **classic** Personal Access Token with `public_repo` scope as a
  repository secret (for example `WINGET_TOKEN`) and pass it to the GoReleaser
  step. The default `GITHUB_TOKEN` cannot open a cross-repository pull request
  to Microsoft's repository.

#### First-Submission Bootstrap

`microsoft/winget-pkgs` is a curated community repository: every submission is a
pull request gated by automated validation **and** human moderator review, so
publication is not self-controlled the way the hand-maintained Homebrew tap is.
The first version for a brand-new package receives extra review scrutiny. The
initial submission may be performed manually with `wingetcreate` and subsequent
version bumps left to the automated GoReleaser PR, or the first PR may be opened
by GoReleaser directly — this task owns choosing and documenting the bootstrap
path.

#### Documentation

Update `docs/workflow/releasing.md` with the WinGet step: the expected Windows
release assets, the `WINGET_TOKEN` secret, the fork prerequisite, and the
moderation-review latency, so release timing accounts for it.

Non-goals: no MSI/NSIS installer, no self-hosted WinGet source, and no coupling
of WinGet publication to anything other than the existing tag-triggered release.

## Caution

Do not turn `v0.3.0` into:

- a generic PM system,
- a hosted planning platform,
- a giant agent framework.

Keep it grounded in repo-native files, explicit structure, inspectable outputs,
and human reviewability.

## Recommendation About LLM Support

Unchanged from `v0.2.0`. Do not add direct LLM calls into Taskrail core.
Preferred order remains: manual CLI and file model; prompts and skills for
coding agents; an optional thin adapter layer only later, only if still needed.

## Explicitly Excluded

Continue to exclude:

- deep semantic drift detection,
- automatic planning loops,
- background daemons,
- complex multi-agent runtime logic,
- a skill execution, scheduling, or orchestration runtime inside Taskrail,
- repair actions that advance task status or fabricate work,
- historical/time-series statistics and any persistent event log backing them,
- promoting manual-test artifacts to a required product invariant.
