# Taskrail v0.1.0

## Summary

Taskrail `v0.1.0` is a minimal viable product focused on bootstrapping into an existing repository, explicit repo structure, deterministic next-task selection, an authoritative `planning/STATE.md`, minimal task transitions, verification artifacts, follow-up task creation, and spec linkage in tasks.

This version is manual-first and LLM-provider-agnostic.

## Goals

- Prove the Taskrail repository contract.
- Prove deterministic task progression.
- Prove explicit state continuity in `planning/STATE.md`.
- Prove verification as a first-class concept.
- Be easy to retrofit into existing repos.
- Be usable manually by humans and coding agents.

## Commands

### taskrail init

Initialize Taskrail inside an existing repository.

It should create:

- `specs/`
- `planning/`
- `planning/tasks/`
- `planning/artifacts/`
- starter `specs/README.md`
- starter `planning/STATE.md`

Why:

- Retrofit into existing repos is a key early use case.

### taskrail validate

Validate:

- folder layout
- required files
- task file shape
- dependency references
- spec references
- `STATE.md` consistency

Why:

- Humans and agents need one command that tells them whether the repo is in a sane state.

### taskrail next

Deterministically select the next eligible task based on:

- status
- dependencies
- priority
- stable tie-breaking

Why:

- Without this, Taskrail is mostly markdown conventions.

### taskrail start <task-id>

Mark a task as active and update `planning/STATE.md`.

Why:

- This establishes explicit current work.

### taskrail complete <task-id>

Mark a task as completed from an implementation perspective.

Why:

- Work needs to advance while verification stays distinct.

### taskrail block <task-id>

Mark a task as blocked and capture a reason.

Why:

- Blocking is a normal real-world outcome and should be first-class.

### taskrail verify <task-id>

Record verification outcome and write an artifact under `planning/artifacts/verify/`.

Why:

- Verification is core to the product concept.

## Follow-Up Task Creation

Taskrail should allow verification or a simple manual flow to create follow-up tasks when additional work is discovered.

Why:

- Real work often reveals missing steps, regressions, or unresolved issues.

## Deterministic Task Schema

Task files should support:

- stable task IDs
- title
- status
- priority
- dependencies
- spec reference

Why:

- This is foundational for deterministic behavior.

## STATE Contract

One authoritative state file must exist and be used as the continuity and control surface.

Why:

- This is one of the core concepts of Taskrail.

## Explicitly Excluded

Do not include in `v0.1.0`:

- `loop`
- import or retrofit content generation
- built-in LLM calls
- spec-to-task auto-generation
- spec-task semantic drift detection
- background daemons
- multiple lanes
- concurrency or distributed workers
- worktree, sandbox, or orchestration features

## Implementation Bias

Prefer:

- simple CLI
- predictable file operations
- human-readable markdown
- minimal dependencies
- easy manual inspection

Avoid:

- prematurely abstract architecture
- hidden automation
- too many commands
- overcomplicated schemas
