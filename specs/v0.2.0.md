# Taskrail v0.2.0

## Summary

Taskrail `v0.2.0` builds on the stable `v0.1.0` repo contract and adds retrofitting and import support, while still avoiding direct LLM-provider integration in the Taskrail CLI itself.

The design should support:

- manual CLI usage
- coding-agent skills and prompts
- helper scripts
- structured import workflows

## Goals

- Make Taskrail easier to adopt in existing repositories with existing notes, specs, and ideas.
- Support transforming rough human input into Taskrail structure.
- Keep provider and tooling independence.
- Avoid coupling the core product to one LLM provider.

## Potential Features

### taskrail import

Import source material into Taskrail structure.

Potential sources:

- notes markdown files
- rough feature docs
- bug notes
- todo docs
- draft specs

Potential outputs:

- spec drafts
- task drafts
- combined planning bootstrap

Possible usage patterns:

- `taskrail import notes.md --to tasks`
- `taskrail import feature.md --to spec`
- `taskrail import notes.md --to planning`

This can initially be scaffolded to work with manual prompts and skills instead of built-in LLM calls.

### Guided Retrofit Bootstrap Flow

A flow that helps take an existing repo and some human notes and turn them into:

- `specs/`
- `planning/tasks/`
- initial `STATE.md`

This can begin as a command, a script, or a documented workflow with skills first.

### Better Task Creation Ergonomics

Potential additions:

- templates
- task scaffolding
- more ergonomic follow-up generation

### Better Validation Around Spec And Task Linkage

At this stage, add stronger checks such as:

- referenced spec exists
- task links are valid
- task points to known spec sections or IDs when present

### Agent Workflow Skills

Taskrail should provide a small, reusable set of coding-agent skills that
encode the tracked-work workflow (validate, select, start, implement, verify,
follow-up, recover) so agents in any adopting repository can drive Taskrail
consistently.

These skills already exist in this repository as temporary dogfooding
scaffolding under `skills/`, mirrored into `.agents/skills/` and
`.claude/skills/`. v0.2.0 should decide how, if at all, they become a supported
part of the product rather than repo-local bootstrap material.

Three decisions must be made explicitly:

- **Portability.** Shipped skills must be repo-agnostic. They must invoke the
  installed `taskrail` binary, not `go run ./cmd/taskrail`, which only works
  inside the Taskrail source tree. This repository's own dogfooding skills may
  keep `go run` until the binary is the dogfooding entry point.
- **Distribution mechanism.** Pick exactly one initial path and document it:
  - documentation-only (publish skill text; adopters copy manually), or
  - opt-in install via `taskrail init --with-skills`, or
  - embedded skill files (`embed.FS`) written on an explicit opt-in.
  The default `taskrail init` must not silently write agent-tool directories.
- **Relationship to task-creation ergonomics.** Today the skills compensate for
  the absence of a standalone task-creation command by hand-authoring task
  files. The "Better Task Creation Ergonomics" work above reduces what the
  skills must carry; the two should be designed together so a skill can call a
  real command instead of writing markdown by hand.

Taskrail distributes skills as static, provider-agnostic text. It does not
execute, schedule, or orchestrate them. Running a skill remains the agent's
responsibility, consistent with the LLM and runtime exclusions below.

### State Repair

`taskrail validate` detects inconsistencies but offers no guided way to fix
them. The current `autonomous-recovery` skill falls back to explicit manual
edits of authoritative state, which the workflow rules otherwise forbid.

v0.2.0 should consider a conservative repair surface, for example
`taskrail repair` (or `taskrail validate --fix`), that:

- proposes the minimal safe correction for each detected violation,
- defaults to a dry run and shows the diff before applying,
- only repairs mechanical inconsistencies (for example stale task counts or a
  `STATE.md` field that disagrees with task files),
- never advances status or invents work to force progress,
- re-runs validation after applying and reports the result.

This keeps state authoritative and removes the recovery skill's need to bypass
the CLI.

### Artifact And Init Consistency

`taskrail init` currently creates `planning/artifacts/verify/` and
`planning/artifacts/runs/`, but the manual-test workflow writes under
`planning/artifacts/manual-test/`, which init does not create.

v0.2.0 should resolve this one way or the other:

- standardize a documented artifacts subdirectory convention that init
  provisions, or
- keep manual-test artifacts as an internal dogfooding convention and leave
  them out of the product surface.

Verification artifacts remain the first-class, product-level concept either
way; manual-test artifacts should not be promoted to a product invariant
without a deliberate decision.

#### Validate Must Not Require Gitignored Artifact Directories

`taskrail validate` currently requires `planning/artifacts` and
`planning/artifacts/verify` to *exist* as part of its structural checks. But
those directories are gitignored output (see "Portable Committed State" below),
and `taskrail init` scaffolds them empty. Git cannot track empty directories, so
a normal commit/clone drops them, and a freshly init-ed and committed repository
then fails `validate` on a clean checkout (fresh clone or CI) for reasons
unrelated to planning content. Under v0.2.0 this is permanent, not transient:
because artifacts are deliberately gitignored, `planning/artifacts/verify/` is
empty on every clean checkout, so a "this directory must exist" check can never
be satisfied by committed content.

v0.2.0 resolves this: **`validate` must not require gitignored artifact output
directories to exist.** A missing `planning/artifacts/` or
`planning/artifacts/verify/` means "no artifacts produced yet" and is valid;
`validate` inspects their contents only when present. `taskrail verify`
continues to create the artifact tree on demand when it writes evidence, so no
directory needs to pre-exist. No placeholder files (`.gitkeep`) are introduced —
that would contradict the gitignored-artifacts decision.

This also removes an existing asymmetry: `validate` today requires `artifacts`
and `verify` but not `runs`, even though `init` scaffolds all three. After this
change none of the artifact output directories are required.

This reconciles with "Portable Committed State" into one coherent artifact
contract: the artifacts tree is gitignored, reproducible local output, and
neither committed state nor structural validation depends on it surviving a Git
round-trip.

**Invariant:** a repository that is freshly `taskrail init`-ed and committed
must pass `taskrail validate` on a clean checkout (fresh clone / CI), with no
placeholder files required.

### Portable Committed State

Generated artifacts under `planning/artifacts/` are deliberately gitignored
("generated evidence; keep out of version control"). That decision is sound and
should stand. The defect is that committed files embed pointers into those
uncommitted artifacts. `taskrail verify` writes a timestamped artifact path into
the committed `STATE.md` frontmatter (`last_verification_result`,
`relevant_artifacts`) and appends the same path to the verified task's notes.

A teammate who clones the repository receives `STATE.md` and task files that
reference timestamped, machine-local files which exist only on the producer's
disk. The references resolve nowhere, add no value, and silently rot. Today
`validate` does not flag them, so the rot is invisible.

The contract: anything written into a committed file must be meaningful to
anyone who clones the repository. The verification *result* is first-class and
portable; the raw evidence files are local and reproducible by re-running
`verify`. Reconcile this together with the Artifact And Init Consistency
decision above so there is one coherent artifact contract.

#### Portable Verification Result

`taskrail verify` must record a portable verification result in committed state
rather than a path into a gitignored artifact:

- `last_verification_result` becomes a portable summary (at least task id,
  result, and timestamp) with no path into gitignored content.
- `relevant_artifacts` lists only committed paths, or is empty; it must never
  list a gitignored artifact path.
- A producer who wants to locate their own local evidence may use a gitignored
  side-channel (for example under `planning/artifacts/runs/`); committed state
  must not carry that pointer.

#### No Local Paths In Task Notes

The verify writer also appends a timestamped artifact path to the verified
task's notes (`appendTaskNote`). Task files are committed, so this leaks the
same local-only reference. The appended note must record the portable result
without a path into gitignored content.

#### Scrub Existing Dangling References

Existing committed files already contain dangling references (for example
`STATE.md` and `T-017`). After the writers above stop producing them, perform a
one-time scrub of the dangling artifact references already committed so the
repository is clean and no stale local path remains in tracked files.

### Layout Config And Versioned Init Upgrade

Today the repository layout is hardcoded. `DiscoverPaths` walks up to the
nearest `.git` directory and then assumes a fixed layout (`specs/`,
`planning/`, `planning/STATE.md`, `planning/artifacts/`). There is no file that
marks a repository as Taskrail-managed, and no recorded layout version. This
blocks three things v0.2.0 wants: a clear "this repo uses Taskrail" signal,
support for adopting Taskrail into repositories with a different setup, and a
safe path to upgrade an older layout when the contract evolves.

The unifying primitive is a single versioned layout marker. Adding it solves
discoverability and upgrade together. The following decisions scope that work.

#### Layout Config And Marker

Introduce one machine-owned marker file, `.taskrail/config.yml`, that serves
three roles at once:

- the discoverable signal that a repository is Taskrail-managed,
- the migration anchor, via an explicit `layout_version` field,
- the indirection point that records where human-facing content lives.

Human-facing content stays visible and top-level. `specs/` and `planning/` are
PR-reviewed and human-edited; they must not be buried inside a dotfolder, which
would read as "tooling internals, do not touch" and undercut the markdown
contract this project is built around. Only the marker, the layout version, and
machine configuration belong under `.taskrail/`.

Keep the config minimal. Record the layout version and the locations Taskrail
already uses; do not add configuration knobs that nothing reads yet. Arbitrary
path relocation is explicitly deferred until a second real layout exists.

#### Config-Aware Path Discovery

`DiscoverPaths` should, after locating the repository root, look for
`.taskrail/config.yml` and resolve paths from it. When the marker is absent it
must fall back to the current hardcoded v0.1.0 layout unchanged. This keeps
every existing repository working with no migration required and makes the
marker purely additive.

#### Versioned Init And Upgrade

`taskrail init` becomes version-aware rather than the current blind
`ensureDir` / `writeFileIfMissing` pass:

- empty repository: fresh init writes the current layout and the marker,
- existing v0.1.0 layout with no marker: detect it, write a marker pinning the
  current layout, and change nothing else (non-destructive adoption),
- marker present with an older `layout_version`: migrate to the current layout,
  defaulting to a dry run that shows the diff before applying, then re-running
  validation, mirroring the conservative discipline required of `taskrail
  repair`.

Migration must never destroy or rewrite human-authored content and must be
reversible in effect (dry-run first, explicit apply).

#### Guided Retrofit Detection

Extend the retrofit flow so init can analyze a repository that does not match a
known Taskrail layout, detect existing markers or candidate directories, and
propose a mapping for the adopter to confirm. Detection proposes; it never
clobbers existing files without explicit confirmation. This is the
analyze-and-adapt behavior the Guided Retrofit Bootstrap Flow above anticipates,
now made version-aware through the layout marker.

### Mise Toolchain Management

Taskrail has no pinned developer toolchain: contributors install Go, `task`, and
(for git hooks) `lefthook` by hand, and nothing provisions them on a fresh
clone. Sibling repositories standardize on [mise](https://mise.jdx.dev) to pin
and provision the developer environment.

v0.2.0 should adopt mise as the optional, documented way to provision that
environment:

- A committed `mise.toml` pins the toolchain (Go, `task`, `lefthook`) to explicit
  versions, so `mise install` reproduces the environment on a fresh clone.
- Provisioning wires the opt-in git hooks (a mise task that runs
  `lefthook install`), so a single command takes a clean checkout from clone to
  hooks-installed.
- mise stays optional convenience, consistent with `task`: direct `go` commands
  remain canonical, and CI may adopt mise but is not required to.
- The pinned versions are the single source of truth shared by local setup and
  CI (the `lefthook` pin matches `task hooks:install`; the Go pin matches
  `go.mod`).

This is developer-environment tooling only; it does not change Taskrail product
behavior.

## Recommendation About LLM Support

Do not rush to add direct LLM calls into Taskrail core.

Preferred order:

1. manual CLI and file model
2. prompts and skills for coding agents
3. optional thin adapter layer later if still needed

## Explicitly Excluded

Prefer to still exclude:

- deep semantic drift detection
- automatic planning loops
- background daemons
- complex multi-agent runtime logic
- a skill execution, scheduling, or orchestration runtime inside Taskrail
- repair actions that advance task status or fabricate work
