# Taskrail v0.2.0

## Summary

Taskrail `v0.2.0` builds on the stable `v0.1.0` repo contract and adds retrofitting and import support, while still avoiding direct LLM-provider integration in the Taskrail CLI itself.

The design should support:

- manual CLI usage
- coding-agent skills and prompts
- helper scripts
- structured import workflows

## Goals

- Make Taskrail easier to adopt in existing repositories with existing notes, specs, and ideas.
- Support transforming rough human input into Taskrail structure.
- Keep provider and tooling independence.
- Avoid coupling the core product to one LLM provider.

## Potential Features

### taskrail import

Import source material into Taskrail structure.

Potential sources:

- notes markdown files
- rough feature docs
- bug notes
- todo docs
- draft specs

Potential outputs:

- spec drafts
- task drafts
- combined planning bootstrap

Possible usage patterns:

- `taskrail import notes.md --to tasks`
- `taskrail import feature.md --to spec`
- `taskrail import notes.md --to planning`

This can initially be scaffolded to work with manual prompts and skills instead of built-in LLM calls.

### Guided Retrofit Bootstrap Flow

A flow that helps take an existing repo and some human notes and turn them into:

- `specs/`
- `planning/tasks/`
- initial `STATE.md`

This can begin as a command, a script, or a documented workflow with skills first.

### Better Task Creation Ergonomics

Potential additions:

- templates
- task scaffolding
- more ergonomic follow-up generation

### Better Validation Around Spec And Task Linkage

At this stage, add stronger checks such as:

- referenced spec exists
- task links are valid
- task points to known spec sections or IDs when present

### Agent Workflow Skills

Taskrail should provide a small, reusable set of coding-agent skills that
encode the tracked-work workflow (validate, select, start, implement, verify,
follow-up, recover) so agents in any adopting repository can drive Taskrail
consistently.

These skills already exist in this repository as temporary dogfooding
scaffolding under `skills/`, mirrored into `.agents/skills/` and
`.claude/skills/`. v0.2.0 should decide how, if at all, they become a supported
part of the product rather than repo-local bootstrap material.

Three decisions must be made explicitly:

- **Portability.** Shipped skills must be repo-agnostic. They must invoke the
  installed `taskrail` binary, not `go run ./cmd/taskrail`, which only works
  inside the Taskrail source tree. This repository's own dogfooding skills may
  keep `go run` until the binary is the dogfooding entry point.
- **Distribution mechanism.** Pick exactly one initial path and document it:
  - documentation-only (publish skill text; adopters copy manually), or
  - opt-in install via `taskrail init --with-skills`, or
  - embedded skill files (`embed.FS`) written on an explicit opt-in.
  The default `taskrail init` must not silently write agent-tool directories.
- **Relationship to task-creation ergonomics.** Today the skills compensate for
  the absence of a standalone task-creation command by hand-authoring task
  files. The "Better Task Creation Ergonomics" work above reduces what the
  skills must carry; the two should be designed together so a skill can call a
  real command instead of writing markdown by hand.

Taskrail distributes skills as static, provider-agnostic text. It does not
execute, schedule, or orchestrate them. Running a skill remains the agent's
responsibility, consistent with the LLM and runtime exclusions below.

### State Repair

`taskrail validate` detects inconsistencies but offers no guided way to fix
them. The current `autonomous-recovery` skill falls back to explicit manual
edits of authoritative state, which the workflow rules otherwise forbid.

v0.2.0 should consider a conservative repair surface, for example
`taskrail repair` (or `taskrail validate --fix`), that:

- proposes the minimal safe correction for each detected violation,
- defaults to a dry run and shows the diff before applying,
- only repairs mechanical inconsistencies (for example stale task counts or a
  `STATE.md` field that disagrees with task files),
- never advances status or invents work to force progress,
- re-runs validation after applying and reports the result.

This keeps state authoritative and removes the recovery skill's need to bypass
the CLI.

### Artifact And Init Consistency

`taskrail init` currently creates `planning/artifacts/verify/` and
`planning/artifacts/runs/`, but the manual-test workflow writes under
`planning/artifacts/manual-test/`, which init does not create.

v0.2.0 should resolve this one way or the other:

- standardize a documented artifacts subdirectory convention that init
  provisions, or
- keep manual-test artifacts as an internal dogfooding convention and leave
  them out of the product surface.

Verification artifacts remain the first-class, product-level concept either
way; manual-test artifacts should not be promoted to a product invariant
without a deliberate decision.

## Recommendation About LLM Support

Do not rush to add direct LLM calls into Taskrail core.

Preferred order:

1. manual CLI and file model
2. prompts and skills for coding agents
3. optional thin adapter layer later if still needed

## Explicitly Excluded

Prefer to still exclude:

- deep semantic drift detection
- automatic planning loops
- background daemons
- complex multi-agent runtime logic
- a skill execution, scheduling, or orchestration runtime inside Taskrail
- repair actions that advance task status or fabricate work
